## 概述

**内存安全 (Memory Safety)** 是软件安全的核心领域，关注程序如何正确、安全地管理内存。内存安全漏洞（如缓冲区溢出、Use-After-Free）是历史上最常见的安全漏洞类型。


## 缓冲区溢出

缓冲区溢出是最经典的内存安全漏洞，当程序向缓冲区写入超过其容量的数据时发生。

```
┌─────────────────────────────────────────────────────────────────┐
│                   栈缓冲区溢出原理                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   正常栈帧布局:                                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  高地址                                                  │   │
│   │  ┌─────────────────┐                                    │   │
│   │  │  函数参数        │                                    │   │
│   │  ├─────────────────┤                                    │   │
│   │  │  返回地址 (RA)   │  ← 函数执行完后跳转到此处           │   │
│   │  ├─────────────────┤                                    │   │
│   │  │  局部变量[0-15]  │  ← buffer[16]                     │   │
│   │  └─────────────────┘                                    │   │
│   │  低地址                                                  │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   溢出后的栈帧:                                                   │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  高地址                                                  │   │
│   │  ┌─────────────────┐                                    │   │
│   │  │  函数参数        │                                    │   │
│   │  ├─────────────────┤                                    │   │
│   │  │  恶意代码地址    │  ← 被覆盖的返回地址                 │   │
│   │  ├─────────────────┤                                    │   │
│   │  │  AAAAA...Shellcd │  ← 填充数据 + 恶意代码             │   │
│   │  └─────────────────┘                                    │   │
│   │  低地址                                                  │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**漏洞代码示例：**

```c
// 不安全的代码 - 存在缓冲区溢出
#include <stdio.h>
#include <string.h>

void vulnerable_function(char *input) {
    char buffer[64];
    strcpy(buffer, input);  // 危险: 不检查长度
}

// 安全的修复
void secure_function(char *input) {
    char buffer[64];
    strncpy(buffer, input, sizeof(buffer) - 1);
    buffer[sizeof(buffer) - 1] = "\0";
}
```

## ASLR 和 DEP/NX

现代系统使用多种防护机制阻止内存攻击。

```
┌─────────────────────────────────────────────────────────────────┐
│                   内存保护机制                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ASLR (Address Space Layout Randomization):                    │
│   • 随机化栈、堆、库的位置                                       │
│   • 增加攻击者预测地址的难度                                     │
│   • Linux: /proc/sys/kernel/randomize_va_space                  │
│                                                                 │
│   DEP/NX (Data Execution Prevention / No-Execute):              │
│   • 标记数据页为不可执行                                         │
│   • 阻止在栈/堆上执行代码                                        │
│   • x86 CPU通过NX bit实现                                        │
│                                                                 │
│   Stack Canary:                                                 │
│   • 在返回地址前插入随机值                                        │
│   • 函数返回前检查该值是否被修改                                  │
│   • GCC: -fstack-protector-strong                               │
│                                                                 │
│   RELRO (Relocation Read-Only):                                 │
│   • 保护GOT表不被覆盖                                            │
│   • Full RELRO: 完全只读                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**编译器保护选项：**

```bash
# 安全编译选项
$ gcc -o secure_app source.c \
    -fstack-protector-strong \
    -D_FORTIFY_SOURCE=2 \
    -fPIE -pie \
    -Wl,-z,relro,-z,now \
    -Wl,-z,noexecstack

# 检查二进制保护
$ checksec --file=./secure_app
RELRO:    Full RELRO
Stack:    Canary found
NX:       NX enabled
PIE:      PIE enabled
```

## Use-After-Free

堆内存管理错误是另一类严重的内存安全漏洞。

```
┌─────────────────────────────────────────────────────────────────┐
│                   Use-After-Free 原理                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   正常内存生命周期:                                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  1. malloc()  ──▶  分配内存                              │   │
│   │  2. 使用内存                                              │   │
│   │  3. free()    ──▶  释放内存                              │   │
│   │  4. 内存返回给堆管理器                                     │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   Use-After-Free 漏洞:                                            │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  1. ptr = malloc(100)                                     │   │
│   │  2. free(ptr)    ← 释放内存                              │   │
│   │  3. 忘记设置 ptr = NULL                                   │   │
│   │  4. new_ptr = malloc(100)  ← 可能获得同一内存块          │   │
│   │  5. *new_ptr = attacker_data                             │   │
│   │  6. *ptr  ← 使用悬空指针! 访问攻击者数据                 │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 安全编程语言

现代安全语言通过编译时检查防止内存安全问题。

```
┌─────────────────────────────────────────────────────────────────┐
│                   安全语言对比                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Rust 内存安全模型:                                              │
│   • 所有权系统 (Ownership): 每个值只有一个所有者                 │
│   • 借用规则: 同时只能有一个可变引用或多个不可变引用              │
│   • 生命周期: 编译时验证引用有效性                               │
│                                                                 │
│   Go 内存安全:                                                    │
│   • 垃圾回收器自动管理内存                                       │
│   • 切片越界运行时检查                                           │
│   • nil指针解引用会panic                                         │
│                                                                 │
│   Swift 内存安全:                                                 │
│   • ARC (Automatic Reference Counting)                          │
│   • 可选类型防止nil解引用                                        │
│   • 数组越界检查                                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Rust 内存安全示例：**

```rust
// Rust在编译期防止内存错误
fn main() {
    // 所有权系统防止Use-After-Free
    {
        let s = String::from("hello");
    }  // s在这里被drop，自动释放
    
    // println!("{}", s);  // 编译错误! s已无效
    
    // 借用检查器防止数据竞争
    let mut data = vec![1, 2, 3];
    let ref1 = &data;  // 不可变借用
    // let ref2 = &mut data;  // 编译错误! 不能同时有可变借用
}
```

## 内存 Sanitizers

Sanitizers 是编译器工具，用于检测运行时的内存错误。

```
┌─────────────────────────────────────────────────────────────────┐
│                   Sanitizer 工具对比                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   AddressSanitizer (ASan):                                      │
│   • 检测: 缓冲区溢出、UAF、Double Free、内存泄漏                │
│   • 开销: 约2-3x内存，2x运行时间                                │
│   • 使用: -fsanitize=address                                    │
│                                                                 │
│   MemorySanitizer (MSan):                                       │
│   • 检测: 使用未初始化内存                                       │
│   • 开销: 约2-3x内存，3x运行时间                                │
│   • 使用: -fsanitize=memory                                     │
│                                                                 │
│   ThreadSanitizer (TSan):                                       │
│   • 检测: 数据竞争                                              │
│   • 开销: 约5-15x内存和时间                                     │
│   • 使用: -fsanitize=thread                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Sanitizer 使用：**

```bash
# 编译时使用ASan
$ gcc -g -fsanitize=address -o test_asan test.c

# 运行，ASan会自动报告错误
$ ./test_asan

# 检测内存泄漏
$ ASAN_OPTIONS=detect_leaks=1 ./leak_test
```

## 面试要点

**Q1: 什么是缓冲区溢出？如何防护？**
> 缓冲区溢出是向缓冲区写入超过其容量的数据，可能覆盖返回地址、函数指针等。防护措施：1) 使用安全函数检查边界；2) 编译器启用Stack Canary、PIE；3) 系统启用ASLR、DEP/NX；4) 使用安全语言替代。

**Q2: ASLR和DEP的关系？**
> DEP阻止在数据区执行代码，迫使攻击者使用ROP；ASLR随机化代码位置，使ROP攻击难以定位gadget。两者结合显著提高攻击难度。

**Q3: Rust如何保证内存安全？**
> Rust通过所有权系统和借用检查器在编译期保证内存安全：1) 每个值只有一个所有者；2) 同时只能有一个可变引用或多个不可变引用；3) 生命周期系统确保引用有效。这些检查在编译期完成，无运行时开销。
