# CAP/BASE理论

**CAP定理**（又称**布鲁尔定理, Brewer's Theorem**）是分布式系统领域的核心理论，由Eric Brewer于2000年提出。它指出：在一个分布式系统中，**一致性（Consistency）**、**可用性（Availability）**、**分区容错性（Partition Tolerance）**这三个属性**不可能同时满足**，最多只能同时满足其中两个。

```
CAP三角:

       一致性 (Consistency)
            /\
           /  \
          /    \
         /      \
        /   CAP   \
       /    定理    \
      /______________\
分区容错性          可用性
(Partition          (Availability)
 Tolerance)
```

## CAP三要素

### 1. 一致性 (Consistency)

**定义**: 所有节点在同一时间看到的数据是一致的。读操作总能返回最新的写操作结果。

**特点**:
- 强一致性：每次读取都返回最新写入的值
- 弱一致性：可能读取到旧值，但最终会一致
- 最终一致性：不保证立即一致，但保证最终会一致

```
一致性的要求:

客户端A          节点1          节点2          客户端B
  │    写入x=1    │              │              │
  │──────────────>│              │              │
  │               │   同步x=1    │              │
  │               │─────────────>│              │
  │               │   确认OK     │              │
  │    写入成功   │<─────────────│              │
  │<──────────────│              │              │
  │                              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回x=1    │
  │                              │─────────────>│
  │                              │              │
  必须读取到最新值x=1
```

### 2. 可用性 (Availability)

**定义**: 每个请求都能在合理的时间内获得非错误的响应，但不保证返回的是最新数据。

**特点**:
- 系统总是可用，不返回错误
- 不保证数据是最新的
- 部分节点故障时仍能提供服务

```
可用性的要求:

客户端A          节点1(故障)      节点2          客户端B
  │    写入x=1    │     X        │              │
  │──────────────>│    故障      │              │
  │               │              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回x=0    │
  │                              │─────────────>│
  │                              │              │
  即使节点1故障，节点2仍能响应
  (可能返回旧值x=0，但不是错误)
```

### 3. 分区容错性 (Partition Tolerance)

**定义**: 系统在遇到网络分区（节点间通信中断）时仍能继续运行。

**特点**:
- 网络分区不可避免
- 分布式系统必须容忍分区
- 分区时需要在C和A之间做选择

```
网络分区场景:

    节点A  <--❌网络中断❌-->  节点B
      │                           │
      │ 分区1                      │ 分区2
      │                           │
    客户端1                       客户端2
      
分区发生后:
- 如果保证一致性(C)：必须拒绝写入，牺牲可用性(A)
- 如果保证可用性(A)：允许各自写入，牺牲一致性(C)
```

## CAP不可能三角

```
分布式系统必须在C、A、P中选择:

┌─────────────────────────────────────────────────────┐
│                                                     │
│    ┌─────────┐      ┌─────────┐      ┌─────────┐   │
│    │   CP    │      │   AP    │      │   CA    │   │
│    │ 系统    │      │ 系统    │      │ 系统    │   │
│    └────┬────┘      └────┬────┘      └────┬────┘   │
│         │                │                │        │
│    一致+分区容错    可用+分区容错    一致+可用       │
│         │                │                │        │
│    牺牲可用性       牺牲一致性       牺牲分区容错    │
│         │                │                │        │
│    ZooKeeper       Cassandra       单机MySQL      │
│    etcd            DynamoDB        单节点Redis    │
│    HBase           Eureka                            │
│                                                     │
│  ⚠️ CA在分布式系统中理论上不可能，                 │
│     因为网络分区不可避免                            │
└─────────────────────────────────────────────────────┘
```

### CP系统 (Consistency + Partition Tolerance)

**牺牲可用性，保证一致性和分区容错**

| 系统 | 特点 |
|------|------|
| **etcd** | 强一致性，用于配置管理 |
| **ZooKeeper** | 协调服务，强一致性 |
| **Consul** | 服务发现，CP模式 |
| **MongoDB (配置为CP)** | 主从复制，可配置 |
| **HBase** | 基于HDFS，强一致性 |

**适用场景**:
- 金融交易系统
- 库存管理
- 配置中心
- 对一致性要求极高的场景

```
CP系统行为:

客户端A          主节点          从节点(分区)      客户端B
  │    写入x=1    │              │     X        │
  │──────────────>│              │   分区       │
  │               │   同步失败   │              │
  │               │────❌───────>│              │
  │    写入失败   │              │              │
  │<──────────────│              │              │
  │   (拒绝写入)   │              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回旧值   │
  │                              │   或超时     │
```

### AP系统 (Availability + Partition Tolerance)

**牺牲一致性，保证可用性和分区容错**

| 系统 | 特点 |
|------|------|
| **Cassandra** | 最终一致性，高可用 |
| **DynamoDB** | 亚马逊KV存储，AP模式 |
| **Couchbase** | 分布式文档数据库 |
| **Riak** | 分布式KV存储 |
| **Eureka** | 服务发现，AP模式 |

**适用场景**:
- 社交网络
- 内容分发
- 实时分析
- 对可用性要求极高的场景

```
AP系统行为:

客户端A          节点A(分区1)    节点B(分区2)      客户端B
  │    写入x=1    │              │              │
  │──────────────>│              │              │
  │    写入成功   │              │              │
  │<──────────────│              │              │
  │    (继续服务)  │     X        │              │
  │               │   网络分区    │              │
  │                              │    写入x=2    │
  │                              │<─────────────│
  │                              │    写入成功   │
  │                              │─────────────>│
  │                              │              │
  恢复后需要解决冲突:x=1 vs x=2
```

### CA系统 (Consistency + Availability)

**理论上存在，实际分布式系统中不可能**

- 单机系统可以实现CA
- 分布式系统必须有P（分区容错）
- 因为网络分区不可避免

```
CA系统的局限:

┌─────────────────────────────────────┐
│  单机数据库 (如MySQL单节点)          │
│  - 一致性: ✓ 没有复制延迟           │
│  - 可用性: ✓ 单机正常即可用         │
│  - 分区容错: ✗ 单机无网络问题       │
└─────────────────────────────────────┘
         ↓
    不是真正的分布式系统
```

## BASE理论

**BASE**是**Basically Available, Soft state, Eventually consistent**的缩写，是对CAP中AP方案的延伸，是NoSQL数据库常遵循的设计原则。

```
BASE vs ACID:

┌─────────────────────────────────────────────────────┐
│                                                     │
│   ACID                     BASE                     │
│   ────                     ────                     │
│                                                     │
│   原子性(A)    ──────▶    基本可用(BA)              │
│   Atomicity               Basically Available       │
│                                                     │
│   一致性(C)    ──────▶    软状态(SS)                │
│   Consistency             Soft State                │
│                                                     │
│   隔离性(I)    ──────▶    最终一致(E)               │
│   Isolation               Eventually Consistent     │
│                                                     │
│   持久性(D)                                         │
│   Durability                                        │
│                                                     │
│   强一致性模型 ──────▶    弱一致性模型              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### BA - Basically Available (基本可用)

**定义**: 系统出现故障时，允许损失部分可用性，但核心功能仍然可用。

```
基本可用的体现:

┌─────────────────────────────────────────────────────┐
│                                                     │
│  正常可用                基本可用                   │
│  ────────                ────────                   │
│                                                     │
│  响应时间: 100ms   ──▶   响应时间: 1~2秒(延长)      │
│                                                     │
│  支持1万并发       ──▶   支持8千并发(降级)          │
│                                                     │
│  全功能可用        ──▶   核心功能可用(限流)         │
│                                                     │
│  例: 电商系统故障时，                                 │
│      仍可浏览商品，但暂停下单功能                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### S - Soft State (软状态)

**定义**: 允许系统中的数据存在中间状态，不影响系统整体可用性。

```
软状态的含义:

┌─────────────────────────────────────────────────────┐
│                                                     │
│  传统ACID:              BASE软状态:                 │
│  数据必须时刻一致       数据可以暂时不一致           │
│                                                     │
│  例: 转账操作            例: 用户积分更新             │
│                                                     │
│  A账户: 1000            主库: 积分=100              │
│  B账户: 500             从库A: 积分=100(已同步)     │
│     ↓                   从库B: 积分=80(延迟同步)    │
│  A→B转200                                         │
│     ↓                   短暂不一致是允许的           │
│  A账户: 800                                         │
│  B账户: 700             最终所有从库都会变为100     │
│                                                     │
│  任何时刻读取           读取可能得到不同值            │
│  都必须看到转账结果     但最终会一致                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### E - Eventually Consistent (最终一致性)

**定义**: 不保证立即一致，但保证在没有新更新的情况下，最终所有副本都会达到一致状态。

```
最终一致性的时间线:

  t0        t1        t2        t3        t4
  │         │         │         │         │
  ▼         ▼         ▼         ▼         ▼
┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐
│ A=0│    │ A=1│    │ A=1│    │ A=1│    │ A=1│  ← 主节点
└────┘    └────┘    └────┘    └────┘    └────┘
   │         │         │         │         │
   │  写入A=1 │         │         │         │
   │─────────>│         │         │         │
   │          │  同步   │  同步   │         │
   │          │────────>│────────>│         │
   │          │         │         │         │
┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐
│ B=0│    │ B=0│    │ B=1│    │ B=1│    │ B=1│  ← 副本1
└────┘    └────┘    └────┘    └────┘    └────┘
   │                              ↑
   │                         达到一致
   │
┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐
│ C=0│    │ C=0│    │ C=0│    │ C=1│    │ C=1│  ← 副本2
└────┘    └────┘    └────┘    └────┘    └────┘
                                          ↑
                                     达到一致

不一致窗口: [t0, t3]
最终一致性: t3后所有副本一致
```

## CAP/BASE在实际系统中的应用

### 数据库系统的选择

```
不同数据库的CAP选择:

关系型数据库 (MySQL/PostgreSQL):
├─ 单节点: CA
├─ 主从同步: CP (同步复制) 或 AP (异步复制)
└─ 特点: 强事务支持，适合金融场景

NoSQL数据库:
├─ MongoDB: CP为主，可配置
├─ Cassandra: AP为主，可调一致性级别
├─ Redis: CP (主从同步)
├─ DynamoDB: AP (亚马逊设计)
└─ 特点: 灵活的一致性级别，适合大规模分布式

NewSQL数据库 (TiDB/CockroachDB):
├─ 目标: 提供SQL的便利性 + NoSQL的扩展性
├─ 一致性: 强一致性 (Raft协议)
└─ 特点: CP系统，兼顾扩展性和一致性
```

### 一致性级别的调节

```
可调一致性 (以Cassandra为例):

写一致性级别:
├─ ONE: 写入一个节点即返回 (最快)
├─ QUORUM: 写入大多数节点 (平衡)
└─ ALL: 写入所有节点 (最安全)

读一致性级别:
├─ ONE: 读一个节点 (最快)
├─ QUORUM: 读大多数节点 (平衡)
└─ ALL: 读所有节点 (最准确)

CAP权衡:
┌─────────────┬─────────────┬────────────────┐
│ 写ONE/读ONE │ 写ALL/读ONE │ 写QUORUM/读Q   │
│    AP模式   │    可能不一致 │    CP模式      │
│  最高可用   │             │  强一致性      │
└─────────────┴─────────────┴────────────────┘
```

## 面试要点

### Q1: 为什么分布式系统必须有P?

**答案**:
1. 网络分区在分布式系统中**不可避免**
2. 网络延迟、丢包、节点故障都会导致分区
3. 如果不能容忍分区，系统就退化为单机
4. 因此分布式系统实际上是在**C和A之间选择**

### Q2: BASE和ACID的区别?

**答案**:

| 特性 | ACID | BASE |
|------|------|------|
| 一致性 | 强一致性 | 最终一致性 |
| 可用性 | 故障时不可用 | 基本可用 |
| 性能 | 较低(锁开销) | 较高(无锁) |
| 适用 | 金融交易 | 社交网络 |
| 代表 | MySQL | Cassandra |

### Q3: 如何根据业务场景选择CP还是AP?

**答案**:
```
                    业务场景
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
    金融交易       社交网络        配置中心
    库存扣减       点赞计数        服务发现
        │             │             │
        ▼             ▼             ▼
    必须CP          可以AP          通常CP
    (不能超卖)      (可以不一致)     (必须一致)
        │             │             │
        └─────────────┴─────────────┘
                      │
              选择相应的系统
```

### Q4: 最终一致性的实现方式?

**答案**:
```
实现最终一致性的常用方法:

1. 读修复 (Read Repair)
   读取时检查各副本，修复不一致

2. 反熵 (Anti-entropy)
   后台进程定期比对和同步副本

3. 暗示移交 (Hinted Handoff)
   临时存储无法到达节点的写操作

4. Merkle Tree
   高效比较和同步大量数据
```

### Q5: CAP与PACELC的关系?

**答案**:
**PACELC**是CAP的扩展:

> 如果有分区(**P**)，系统必须在可用性(**A**)和一致性(**C**)间选择；
> 否则(**E**lse)，即使没有分区，也必须在延迟(**L**)和一致性(**C**)间选择。

```
PACELC分析:

P(分区发生)
  │
  ├─ 选择A ──> 牺牲一致性，保证可用性
  │
  └─ 选择C ──> 牺牲可用性，保证一致性

E(Else - 无分区)
  │
  ├─ 选择L ──> 低延迟，可能读到旧数据
  │
  └─ 选择C ──> 强一致性，可能高延迟
```
