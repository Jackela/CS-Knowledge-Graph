# CPU

**CPU**（Central Processing Unit，中央处理器）是计算机的核心组件，负责执行程序指令、处理数据、控制协调计算机各部件工作。它被称为计算机的"大脑"，是现代计算机系统中最重要的硬件之一。

```

┌─────────────────────────────────────────┐
│           应用程序                       │
│     (浏览器、编辑器、游戏等)              │
├─────────────────────────────────────────┤
│           操作系统                       │
│     (Windows、Linux、macOS)             │
├─────────────────────────────────────────┤
│           硬件抽象层                     │
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────┐   │
│  │         CPU 中央处理器           │   │
│  │  ┌─────┐ ┌─────┐ ┌───────────┐ │   │
│  │  │控制 │ │运算 │ │  寄存器组  │ │   │
│  │  │单元 │ │单元 │ │           │ │   │
│  │  └──┬──┘ └──┬──┘ └─────┬─────┘ │   │
│  │     └───────┴──────────┘       │   │
│  │              │                 │   │
│  │         指令流水线              │   │

└─────────────────────────────────────────┘
```

### 1. 控制单元 (Control Unit, CU)

**功能**: 控制协调CPU各部件工作，解释指令、发出控制信号。

```
控制单元工作流程:

取指阶段:
┌─────────┐      ┌─────────┐      ┌─────────┐
│   PC    │─────→│  内存   │─────→│   IR    │
│(程序计数器)     │(取指令)  │      │(指令寄存器)│
└─────────┘      └─────────┘      └─────────┘
     │                                 │
     └─────────────┬───────────────────┘
                   ▼
            ┌─────────────┐
            │   指令译码   │
            │  (识别操作)  │
            └──────┬──────┘
                   ▼
            ┌─────────────┐
            │  控制信号生成 │
            │  (发送命令)  │
            └─────────────┘

主要寄存器:
- PC (Program Counter): 存放下一条指令地址
- IR (Instruction Register): 存放当前执行指令
- MAR (Memory Address Register): 内存地址寄存器
- MDR (Memory Data Register): 内存数据寄存器
```

### 2. 运算单元 (ALU - Arithmetic Logic Unit)

**功能**: 执行算术运算（加减乘除）和逻辑运算（与或非异或）。

```
ALU运算类型:

算术运算:                    逻辑运算:
┌─────────┐                 ┌─────────┐
│  加法   │  5 + 3 = 8      │  AND    │  1100     │
│  减法   │  5 - 3 = 2      │         │& 1010     │
│  乘法   │  5 × 3 = 15     │         │= 1000     │
│  除法   │  5 ÷ 3 = 1...2  │         │           │
└─────────┘                 │  OR     │  1100     │
                            │         ││ 1010     │
移位运算:                    │         │= 1110     │
┌─────────┐                 │         │           │
│ 左移 << │  0011 → 0110    │  XOR    │  1100     │
│ 右移 >> │  0110 → 0011    │         │^ 1010     │
│ 循环移  │  1100 → 0110    │         │= 0110     │
└─────────┘                 └─────────┘

ALU工作流程:
        操作数A           操作数B
           │               │
           ▼               ▼
    ┌─────────────────────────┐
    │          ALU            │
    │   ┌─────────────────┐   │
    │   │  运算逻辑电路    │   │
    │   │  + & | ^ << >>  │   │
    │   └────────┬────────┘   │
    └────────────┼────────────┘
                 ▼
              运算结果
                 │
        ┌────────┴────────┐
        ▼                 ▼
    结果寄存器         标志寄存器
   (ACC等)           (ZF/SF/CF/OF)
```

## 指令流水线

### 流水线原理

**概念**: 将指令执行过程分解为多个阶段，多条指令重叠执行，提高吞吐率。

```
非流水线 vs 流水线:

非流水线执行 (串行):
时间: ──────────────────────────────────────────→
指令1: [取指][译码][执行][访存][写回]
指令2:                    [取指][译码][执行][访存][写回]
指令3:                                         [取指][译码][执行][访存][写回]

流水线执行 (并行):
时间: ──────────────────────────────────────────→
指令1: [取指][译码][执行][访存][写回]
指令2:      [取指][译码][执行][访存][写回]
指令3:           [取指][译码][执行][访存][写回]
指令4:                [取指][译码][执行][访存][写回]

理想加速比 ≈ 流水线级数
```

### 五级流水线

```
经典MIPS五级流水线:

时钟:   T1     T2     T3     T4     T5     T6     T7     T8
        │      │      │      │      │      │      │      │
指令1: [ IF ] [ ID ] [ EX ] [ MEM][ WB ]
指令2:        [ IF ] [ ID ] [ EX ] [ MEM][ WB ]
指令3:               [ IF ] [ ID ] [ EX ] [ MEM][ WB ]
指令4:                      [ IF ] [ ID ] [ EX ] [ MEM][ WB ]
指令5:                             [ IF ] [ ID ] [ EX ] [ MEM][ WB ]

阶段说明:
┌──────┬─────────┬─────────────────────────────────────┐
│ 阶段 │  英文   │              功能                   │
├──────┼─────────┼─────────────────────────────────────┤
│  IF  │  Fetch  │  取指 - 从内存取出指令              │
│  ID  │ Decode  │  译码 - 解析指令含义，读寄存器      │
│  EX  │Execute  │  执行 - ALU运算或地址计算           │
│ MEM  │ Memory  │  访存 - 读写内存数据                │
│  WB  │Writeback│  写回 - 结果写回寄存器              │
└──────┴─────────┴─────────────────────────────────────┘
```

### 流水线冒险

```
流水线冒险类型:

1. 结构冒险 (Structural Hazard)
   原因: 硬件资源冲突
   
   例: IF和MEM都需要访问内存
   解决: 使用指令缓存和数据缓存分离的哈佛结构

2. 数据冒险 (Data Hazard)
   原因: 指令间数据依赖
   
   例: 
   ADD R1, R2, R3    # R1 = R2 + R3
   SUB R4, R1, R5    # R4 = R1 - R5 (需要等第一条写完)
   
   解决: 数据前递(Forwarding)、流水线停顿(Stall)

3. 控制冒险 (Control Hazard)
   原因: 分支指令改变程序流
   
   例:
   BEQ R1, R2, Label  # 相等则跳转
   ADD R3, R4, R5     # 不确定是否执行
   
   解决: 
   - 分支预测 (Branch Prediction)
   - 延迟槽 (Delay Slot)
   - 冲刷流水线 (Flush)

分支预测示意:

   实际: 分支指令
            │
       ┌────┴────┐
       ▼         ▼
    不跳转      跳转
       │         │
       ▼         ▼
   预测正确   预测错误 → 冲刷流水线，重新取指
```

## 缓存层次结构

### 为什么需要缓存

```
CPU与内存速度差距:

┌────────────────────────────────────────────────────┐
│                                                    │
│  CPU 时钟: 3~5 GHz (周期约 0.3ns)                  │
│       │                                            │
│       │ 等待                                       │
│       ▼                                            │
│  内存访问: 100~300 时钟周期 (约 50~100ns)          │
│       │                                            │
│       │ CPU需要等待! 效率极低                      │
│       ▼                                            │
│  解决方案: 多级缓存                                 │
│                                                    │
└────────────────────────────────────────────────────┘

存储层次金字塔:

        ┌──────┐
        │ 寄存器│  ← 最快，最小，最贵
        │ ~1KB │
       ┌┴──────┴┐
       │  L1缓存 │  ← 32~64KB/core
       │ ~64KB  │     访问: 3~4周期
      ┌┴────────┴┐
      │   L2缓存  │  ← 256KB~1MB/core
      │  ~512KB  │     访问: 10~20周期
     ┌┴──────────┴┐
     │    L3缓存   │  ← 8~64MB 共享
     │   ~32MB    │     访问: 30~50周期
    ┌┴────────────┴┐
    │     内存      │  ← 8~512GB
    │    ~32GB     │     访问: 200+周期
   ┌┴──────────────┴┐
   │    磁盘/SSD     │  ← 最慢，最大，最便宜
   │    ~1TB+       │     访问: 百万周期
   └────────────────┘

越快越贵，越慢越便宜
```

### 各级缓存特性

```
缓存层次详解:

L1 缓存 (Level 1):
┌─────────────────────────────────────────────┐
│  - 分为 L1-I (指令) 和 L1-D (数据)          │
│  - 每核心独享                                 │
│  - 大小: 32~64KB (I) + 32~64KB (D)          │
│  - 延迟: 3~4 时钟周期                        │
│  - 命中率: ~95%                              │
└─────────────────────────────────────────────┘

L2 缓存 (Level 2):
┌─────────────────────────────────────────────┐
│  - 指令数据统一                                │
│  - 每核心独享 (或共享)                         │
│  - 大小: 256KB ~ 1MB                         │
│  - 延迟: 10~20 时钟周期                      │
│  - 命中率: ~90%                              │
└─────────────────────────────────────────────┘

L3 缓存 (Level 3):
┌─────────────────────────────────────────────┐
│  - 指令数据统一                                │
│  - 多核心共享                                  │
│  - 大小: 8~64MB (消费级可达数百MB)            │
│  - 延迟: 30~50 时钟周期                      │
│  - 命中率: ~80%                              │
└─────────────────────────────────────────────┘
```

### 缓存映射策略

```
缓存映射方式:

1. 直接映射 (Direct Mapped)
   内存块只能放在缓存的特定位置
   
   地址: [ 标记 | 行索引 | 块内偏移 ]
   
   冲突多，但硬件简单

2. 全相联 (Fully Associative)
   内存块可以放在缓存的任意位置
   
   无冲突，但需要并行比较所有行
   成本高，通常只用于小缓存

3. 组相联 (Set Associative) ★最常用
   缓存分为若干组，每组有n个行 (n路组相联)
   内存块先映射到组，组内任意放置
   
   例: 4路组相联
   ┌─────────────────────────────────┐
   │  组0: [行0] [行1] [行2] [行3]   │
   │  组1: [行4] [行5] [行6] [行7]   │
   │  组2: [行8] [行9] [行A] [行B]   │
   │  ...                            │
   └─────────────────────────────────┘
   
   地址: [ 标记 | 组索引 | 块内偏移 ]
   
   平衡了冲突率和硬件复杂度
```

### 缓存一致性 (MESI协议)

```
多核缓存一致性问题:

核心A:                      核心B:
┌─────────┐                ┌─────────┐
│ L1: x=1 │                │ L1: x=? │  ← 可能读到旧值
└─────────┘                └─────────┘
     │                          │
     └──────────┬───────────────┘
                ▼
          ┌───────────┐
          │  内存: x=1 │
          └───────────┘

MESI协议 (缓存行状态):

┌──────┬─────────────────────────────────────┐
│ 状态 │ 说明                                │
├──────┼─────────────────────────────────────┤
│  M   │ Modified (已修改) - 独占，与内存不同 │
│  E   │ Exclusive (独占) - 与内存一致       │
│  S   │ Shared (共享) - 多核共享，只读       │
│  I   │ Invalid (无效) - 数据已过期         │
└──────┴─────────────────────────────────────┘

状态转换示例:

1. 核心A读x (缓存未命中，内存加载):
   A: I → E (独占)

2. 核心B也读x:
   A: E → S (共享)
   B: I → S (共享)

3. 核心A写x:
   A: S → M (修改)
   B: S → I (无效)  ← 通知B失效

4. 核心B读x (发现I状态，重新加载):
   A: M → S (写回内存，变为共享)
   B: I → S (从内存加载)
```

### 超线程技术 (Hyper-Threading/SMT)

```
超线程原理:

传统单核:              超线程核心:
┌─────────────────┐    ┌─────────────────────────────┐
│  执行资源        │    │      执行资源               │
│  ┌───────────┐  │    │  ┌───────────────────────┐  │
│  │  ALU      │  │    │  │  ALU    FPU   Load   │  │
│  │  FPU      │  │    │  │  Store  Branch ...   │  │
│  │  Load     │  │    │  └───────────────────────┘  │
│  │  Store    │  │    │          │                  │
│  └───────────┘  │    │    ┌─────┴─────┐            │
│        ↑        │    │    ▼           ▼            │
│   一套寄存器    │    │ ┌─────┐     ┌─────┐         │
│   一个PC        │    │ │线程A│     │线程B│         │
└─────────────────┘    │ │寄存器│     │寄存器│         │
                       │ │  PC  │     │  PC  │         │
                       │ └─────┘     └─────┘         │
                       └─────────────────────────────┘
                              逻辑CPU x2

工作原理:
1. 单核内有两套独立的寄存器和PC
2. 共享执行单元 (ALU/FPU等)
3. 当一个线程等待(如缓存未命中)，另一个线程使用执行单元
4. 通常提升20-30%性能，而非翻倍

调度视角:
操作系统看到:           实际物理核心:
┌──────────┐          ┌──────────────────┐
│ CPU 0    │          │  物理核心0        │
│ CPU 1    │ ←──────  │  ├─ 逻辑核心0    │
│ CPU 2    │          │  └─ 逻辑核心1    │
│ CPU 3    │ ←──────  │  物理核心1        │
└──────────┘          │  ├─ 逻辑核心0    │
4个逻辑CPU            │  └─ 逻辑核心1    │
                      └──────────────────┘
```

### NUMA架构

```
NUMA (Non-Uniform Memory Access):

UMA (统一内存访问):          NUMA (非统一内存访问):
┌─────────────────┐         ┌─────────┐ ┌─────────┐
│  内存控制器      │         │ 节点0   │ │ 节点1   │
│     ┌───┐       │         │ ┌─────┐ │ │ ┌─────┐ │
│     │内存│       │         │ │CPU0 │ │ │ │CPU2 │ │
│     └───┘       │         │ │CPU1 │ │ │ │CPU3 │ │
│    /  |  \      │         │ └──┬──┘ │ │ └──┬──┘ │
┌───┐┌───┐┌───┐   │         │ ┌──┴──┐ │ │ ┌──┴──┐ │
│CPU0││CPU1││CPU2│   │         │ │本地 │ │ │ │本地 │ │

                  │         │ └─────┘ │ │ └─────┘ │
所有CPU访问       │         └────┬────┘ └────┬────┘
内存延迟相同      │              └─────┬─────┘
                  │            互联网络 (QPI/UPI)
                  │
                  │  本地内存访问: 快 (~50ns)
                  │  远程内存访问: 慢 (~100ns)
```

## 面试要点

### Q1: 缓存命中率低如何排查?

**答案**:
```
排查步骤:

1. 使用perf/cachegrind等工具分析缓存 miss

2. 常见原因及优化:
   - 数据未对齐 → 对齐到缓存行边界 (64B)
   - 缓存行伪共享 → Padding避免多核竞争
   - 随机访问模式 → 改为顺序访问
   - 数据过大 → 分块处理，提高局部性

3. 代码优化示例:
   // 差的访问模式 (列优先)
   for (i=0; i<N; i++)
     for (j=0; j<N; j++)
       sum += a[j][i];  // 跳跃访问

   // 好的访问模式 (行优先)
   for (i=0; i<N; i++)
     for (j=0; j<N; j++)
       sum += a[i][j];  // 顺序访问，缓存友好
```

### Q2: 什么是流水线 bubbles? 如何减少?

**答案**:
```
流水线 bubbles (气泡) 来源:

1. 数据冒险导致停顿
   ADD R1, R2, R3
   SUB R4, R1, R5  ← 等待R1结果
   
   解决: 数据前递 (Forwarding)

2. 控制冒险导致冲刷
   BEQ ...         ← 分支预测失败
   ...             ← 已取指令作废
   
   解决: 分支预测算法优化

3. 结构冒险资源冲突
   
   解决: 增加功能单元，资源复制
```

### Q3: 为什么L1分指令和数据缓存?

**答案**:
```
哈佛结构的优势:

1. 并行访问
   取指和访存可同时进行
   
   时钟1: 取指令A, 同时读数据X
   时钟2: 译码指令A, 同时取指令B

2. 减少冲突
   指令流和数据流分离，互不干扰

3. 独立优化
   - I-Cache: 只读，设计更简单
   - D-Cache: 需要写策略，更复杂
```

### Q4: 进程切换的CPU开销?

**答案**:
```
进程切换涉及:

1. 保存当前进程状态
   - 寄存器值 → PCB (进程控制块)
   - 程序计数器PC
   - 状态寄存器

2. 地址空间切换
   - 刷新TLB (页表缓存)
   - 切换页表基址

3. 加载新进程状态
   - 从PCB恢复寄存器
   - 跳转到PC执行

4. 缓存影响
   - 新进程的代码/数据不在缓存
   - 产生大量缓存未命中

开销: 通常需要数千到上万时钟周期
优化: 线程切换(轻量)、减少切换频率
```

### Q5: 分支预测失败的代价?

**答案**:
```
分支预测失败代价:

现代CPU流水线深度: 15~20级

预测失败时:
1. 已取的错误指令丢弃
2. 流水线冲刷 (Flush)
3. 从正确地址重新取指
4. 等待流水线重新填满

代价: 15~20个时钟周期 (等于流水线深度)

优化建议:
- 减少分支 (用位运算替代)
- 分支 likely/unlikely 提示
- 将大概率分支放在前面
- 循环展开减少分支次数
```
