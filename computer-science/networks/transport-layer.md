# 传输层 (Transport Layer)

## 简介

**传输层（Transport Layer）**是OSI模型的第4层，负责端到端的通信服务，为应用进程提供逻辑通信。主要协议有**TCP**和**UDP**。

## 核心功能

- **复用/分解**：多个应用共享网络连接
- **可靠传输**（TCP）：差错恢复、按序交付
- **流量控制**：防止发送方淹没接收方
- **拥塞控制**：防止网络过载

## UDP（用户数据报协议）

### 特点

- **无连接**：无需建立连接
- **不可靠**：不保证交付
- **低开销**：头部仅8字节
- **快速**：无握手、无确认

### UDP头部

```
┌───────────────────┬───────────────────┐
│   源端口 (16位)    │   目的端口 (16位)  │
├───────────────────┼───────────────────┤
│   长度 (16位)      │   校验和 (16位)    │
└───────────────────┴───────────────────┘

总长度: 8字节
```

### 应用场景

- DNS查询
- 视频流（实时性 > 可靠性）
- 在线游戏
- VoIP

## TCP（传输控制协议）

### 特点

- **面向连接**：三次握手建立连接
- **可靠传输**：确认、重传、序号
- **按序交付**：数据按发送顺序到达
- **流量控制**：滑动窗口
- **拥塞控制**：慢启动、拥塞避免

### TCP头部

```
┌─────┬─────┬───────────────────────┬───────────────────────┐
│  源端口 │ 目的端口 │                       │
├─────┴─────┴───────────────────────┴───────────────────────┤
│                      序列号 (Seq)                         │
├───────────────────────────────────────────────────────────┤
│                   确认号 (Ack)                            │
├─────┬─────┬───────────┬───────────────────────────────────┤
│ 偏移 │ 保留 │ UAPRSF │        窗口大小                    │
├─────┴─────┴───────────┴───────────────────────────────────┤
│       校验和        │         紧急指针                     │
├───────────────────┴───────────────────────────────────────┤
│                      选项 (可变)                          │
└───────────────────────────────────────────────────────────┘

标志位:
URG: 紧急
ACK: 确认
PSH: 推送
RST: 重置
SYN: 同步
FIN: 结束
```

### 三次握手（建立连接）

```
客户端                    服务器
   │                        │
   │──── SYN, seq=x ───────→│
   │                        │
   │←── SYN, ACK, seq=y ────│
   │     ack=x+1            │
   │                        │
   │──── ACK, ack=y+1 ─────→│
   │                        │
   │◄──── 连接建立 ────────►│
```

### 四次挥手（断开连接）

```
客户端                    服务器
   │                        │
   │──── FIN ──────────────→│
   │←──── ACK ──────────────│
   │                        │
   │←──── FIN ──────────────│
   │──── ACK ──────────────→│
   │                        │
   │◄──── 连接关闭 ─────────►│

TIME_WAIT状态: 等待2MSL确保ACK到达
```

### 滑动窗口

```
发送方窗口:
已确认   已发送未确认   可发送   不可发送
├─────────┼──────────┼───────┼──────────┤
         │←──── 窗口大小 ────→│

接收方窗口:
已接收   可接收   不可接收
├─────────┼───────┼──────────┤
         │←──── 窗口 ────→│

窗口大小由接收方通告（流量控制）
```

### 拥塞控制

```
拥塞窗口 (cwnd) 变化:

cwnd
 │  ╱╲        ╱╲
 │ ╱  ╲      ╱  ╲
 │╱    ╲____╱    ╲____
 └───────────────────────────
  慢启动    拥塞避免   超时
  (指数)    (线性)    (重设)

算法:
1. 慢启动: cwnd指数增长
2. 拥塞避免: cwnd线性增长
3. 快速重传: 收到3个重复ACK立即重传
4. 快速恢复: 不降到慢启动
```

## TCP vs UDP 对比

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 顺序 | 有序 | 无序 |
| 流量控制 | 有 | 无 |
| 拥塞控制 | 有 | 无 |
| 头部大小 | 20字节 | 8字节 |
| 延迟 | 较大 | 小 |
| 适用场景 | HTTP/HTTPS、FTP | DNS、视频、游戏 |

## 面试要点

### Q1: 为什么需要TIME_WAIT？

1. 确保最后一个ACK到达对方
2. 等待迷途分节在网络中消失，防止影响新连接

### Q2: SYN Flood攻击

```
攻击: 大量伪造SYN请求，不完成三次握手

解决:
- SYN Cookies
- SYN Proxy
- 增加SYN队列长度
```

### Q3: TCP粘包/拆包

**原因**: TCP是字节流，不保留消息边界

**解决**:
- 固定长度消息
- 长度前缀
- 特殊分隔符
- 自定义协议
