# 数据库范式 (Database Normalization)

## 简介

**数据库范式 (Database Normalization)** 是关系数据库设计中的一套规范，用于减少数据冗余、避免更新异常，并确保数据完整性。通过将数据库结构分解为更小、更规范的表，范式化提高了数据库的效率和可维护性。

```
┌─────────────────────────────────────────────────────────────┐
│                    范式化过程                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   第一范式 (1NF)  →  消除重复组，确保原子性                   │
│        ↓                                                    │
│   第二范式 (2NF)  →  消除部分函数依赖                         │
│        ↓                                                    │
│   第三范式 (3NF)  →  消除传递函数依赖                         │
│        ↓                                                    │
│   BCNF           →  消除主属性对码的依赖                      │
│        ↓                                                    │
│   第四范式 (4NF)  →  消除多值依赖                             │
│        ↓                                                    │
│   第五范式 (5NF)  →  消除连接依赖                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 为什么需要范式化

### 数据冗余问题

```
┌────────────────────────────────────────────────────────────────┐
│                    未范式化的学生选课表                          │
├──────────┬────────┬──────────┬──────────┬──────────┬───────────┤
│ 学生ID   │ 姓名    │ 系名      │ 系主任    │ 课程号    │ 成绩      │
├──────────┼────────┼──────────┼──────────┼──────────┼───────────┤
│ 001      │ 张三    │ 计算机系  │ 王教授    │ C01      │ 85        │
│ 001      │ 张三    │ 计算机系  │ 王教授    │ C02      │ 90        │
│ 002      │ 李四    │ 数学系    │ 李教授    │ C01      │ 78        │
│ 002      │ 李四    │ 数学系    │ 李教授    │ C03      │ 88        │
└──────────┴────────┴──────────┴──────────┴──────────┴───────────┘
                    ↑ 冗余数据重复存储
```

**存在的问题：**
- **插入异常**：无法单独插入系信息
- **删除异常**：删除学生可能导致系信息丢失
- **更新异常**：修改系主任需要更新多条记录

## 第一范式 (1NF - First Normal Form)

### 定义
所有属性值都是**原子的 (Atomic)**，不可再分。

### 要求
1. 每个列只包含单一值
2. 每个列的值类型一致
3. 每个行唯一（通过主键标识）

```
❌ 违反1NF（有多值属性）
┌──────────┬─────────────────────┐
│ 学生ID   │ 课程                 │
├──────────┼─────────────────────┤
│ 001      │ 数学, 物理, 化学     │  ← 多值属性
└──────────┴─────────────────────┘

✅ 符合1NF
┌──────────┬──────────┐
│ 学生ID   │ 课程      │
├──────────┼──────────┤
│ 001      │ 数学      │
│ 001      │ 物理      │
│ 001      │ 化学      │
└──────────┴──────────┘
```

## 第二范式 (2NF - Second Normal Form)

### 定义
满足1NF，且**消除非主属性对主键的部分函数依赖**。

### 适用场景
仅当主键是**复合主键**（多个属性组成）时才可能违反2NF。

```
❌ 违反2NF（部分依赖）
主键：(学生ID, 课程号)

┌──────────┬──────────┬────────┬──────────┐
│ 学生ID   │ 课程号   │ 成绩    │ 学生姓名  │  ← 姓名只依赖学生ID
├──────────┼──────────┼────────┼──────────┤
│ 001      │ C01      │ 85     │ 张三      │
│ 001      │ C02      │ 90     │ 张三      │  ← 姓名重复
└──────────┴──────────┴────────┴──────────┘
         ↑ 姓名只依赖于学生ID，与课程号无关

分解为：
✅ 选课表（2NF）
┌──────────┬──────────┬────────┐
│ 学生ID   │ 课程号   │ 成绩    │
├──────────┼──────────┼────────┤
│ 001      │ C01      │ 85     │
│ 001      │ C02      │ 90     │
└──────────┴──────────┴────────┘

✅ 学生表（2NF）
┌──────────┬────────┐
│ 学生ID   │ 姓名    │
├──────────┼────────┤
│ 001      │ 张三    │
└──────────┴────────┘
```

## 第三范式 (3NF - Third Normal Form)

### 定义
满足2NF，且**消除非主属性对主键的传递函数依赖**。

### 传递依赖
如果 A → B 且 B → C，则 A → C 是传递依赖。

```
❌ 违反3NF（传递依赖）
主键：学生ID

┌──────────┬────────┬──────────┬──────────┐
│ 学生ID   │ 姓名    │ 系名      │ 系主任    │
├──────────┼────────┼──────────┼──────────┤
│ 001      │ 张三    │ 计算机系  │ 王教授    │
│ 002      │ 李四    │ 计算机系  │ 王教授    │  ← 系主任重复
└──────────┴────────┴──────────┴──────────┘
         ↑ 学生ID → 系名 → 系主任（传递依赖）

分解为：
✅ 学生表（3NF）
┌──────────┬────────┬──────────┐
│ 学生ID   │ 姓名    │ 系名      │
├──────────┼────────┼──────────┤
│ 001      │ 张三    │ 计算机系  │
│ 002      │ 李四    │ 计算机系  │
└──────────┴────────┴──────────┘

✅ 系表（3NF）
┌──────────┬──────────┐
│ 系名      │ 系主任    │
├──────────┼──────────┤
│ 计算机系  │ 王教授    │
└──────────┴──────────┘
```

## BC范式 (BCNF - Boyce-Codd Normal Form)

### 定义
满足3NF，且**消除主属性对码的部分和传递函数依赖**。

### 与3NF的区别
3NF允许主属性依赖于非主属性，BCNF不允许任何属性（包括主属性）依赖于非码属性。

```
❌ 违反BCNF
关系：学生(学生ID, 课程号, 教师)
假设：每个教师只教一门课，一门课可由多个教师教

函数依赖：
- (学生ID, 课程号) → 教师
- 教师 → 课程号

┌──────────┬──────────┬────────┐
│ 学生ID   │ 课程号   │ 教师    │
├──────────┼──────────┼────────┤
│ 001      │ 数学     │ 张老师  │
│ 002      │ 数学     │ 张老师  │
│ 001      │ 物理     │ 李老师  │
└──────────┴──────────┴────────┘

问题：如果张老师改为教物理，需要更新多条记录

分解为BCNF：
✅ 选课表
┌──────────┬────────┐
│ 学生ID   │ 教师    │
├──────────┼────────┤
│ 001      │ 张老师  │
│ 002      │ 张老师  │
│ 001      │ 李老师  │
└──────────┴────────┘

✅ 教师课程表
┌────────┬────────┐
│ 教师    │ 课程号  │
├────────┼────────┤
│ 张老师  │ 数学    │
│ 李老师  │ 物理    │
└────────┴────────┘
```

## 第四范式 (4NF - Fourth Normal Form)

### 定义
满足BCNF，且**消除非平凡的多值依赖**。

### 多值依赖
如果对于每个X值，都有一组Y值与之对应，且这组Y值与关系中的其他属性无关，则存在多值依赖 X →→ Y。

```
❌ 违反4NF（多值依赖）
关系：课程(课程名, 教师, 教材)
假设：一门课程有多个教师，每个教师使用不同的教材

┌──────────┬────────┬──────────┐
│ 课程名   │ 教师    │ 教材      │
├──────────┼────────┼──────────┤
│ 数据库   │ 张老师  │ 教材A     │
│ 数据库   │ 张老师  │ 教材B     │
│ 数据库   │ 李老师  │ 教材A     │
│ 数据库   │ 李老师  │ 教材B     │
└──────────┴────────┴──────────┘

问题：教师和教材之间是多对多关系，产生大量冗余

分解为4NF：
✅ 课程教师表
┌──────────┬────────┐
│ 课程名   │ 教师    │
├──────────┼────────┤
│ 数据库   │ 张老师  │
│ 数据库   │ 李老师  │
└──────────┴────────┘

✅ 课程教材表
┌──────────┬──────────┐
│ 课程名   │ 教材      │
├──────────┼──────────┤
│ 数据库   │ 教材A     │
│ 数据库   │ 教材B     │
└──────────┴──────────┘
```

## 第五范式 (5NF - Fifth Normal Form)

### 定义
满足4NF，且**消除非平凡的连接依赖**。

### 连接依赖
如果关系可以通过连接多个投影恢复，则存在连接依赖。

5NF在实际应用中较少见，主要用于处理复杂的约束关系。

## 范式对比总结

| 范式 | 要求 | 解决的问题 | 实际应用频率 |
|------|------|------------|--------------|
| 1NF | 属性原子性 | 多值属性 | 必须满足 |
| 2NF | 消除部分依赖 | 复合主键冗余 | 通常满足 |
| 3NF | 消除传递依赖 | 非主属性冗余 | 强烈推荐 |
| BCNF | 消除主属性依赖 | 主键设计问题 | 推荐满足 |
| 4NF | 消除多值依赖 | 独立多值属性 | 特定场景 |
| 5NF | 消除连接依赖 | 复杂约束 | 极少使用 |

## 反范式化 (Denormalization)

### 什么是反范式化
**反范式化**是故意违反范式规则，引入冗余以提高查询性能。

### 适用场景

```
┌─────────────────────────────────────────────────────────────┐
│                   反范式化适用场景                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 读多写少：查询频率远高于更新频率                         │
│                                                             │
│  2. 实时性要求：需要快速响应复杂查询                         │
│                                                             │
│  3. 数据仓库：分析型查询，数据相对稳定                       │
│                                                             │
│  4. 缓存需求：避免频繁JOIN操作                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 反范式化技术

```sql
-- 1. 冗余字段
-- 在订单表中冗余存储用户名，避免JOIN
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    user_id INT,
    user_name VARCHAR(50),  -- 冗余字段
    total_amount DECIMAL(10, 2)
);

-- 2. 派生字段
-- 存储计算结果
CREATE TABLE products (
    product_id INT PRIMARY KEY,
    price DECIMAL(10, 2),
    quantity INT,
    total_value DECIMAL(10, 2)  -- price * quantity
);

-- 3. 预聚合表
CREATE TABLE daily_sales_summary (
    date DATE PRIMARY KEY,
    total_orders INT,
    total_revenue DECIMAL(12, 2),
    avg_order_value DECIMAL(10, 2)
);

-- 4. 宽表设计
-- 将相关数据合并到一张表
CREATE TABLE user_profile_wide (
    user_id INT PRIMARY KEY,
    -- 基本信息
    username VARCHAR(50),
    email VARCHAR(100),
    -- 统计信息（冗余）
    order_count INT DEFAULT 0,
    total_spent DECIMAL(12, 2) DEFAULT 0,
    last_order_date DATE
);
```

### 维护冗余数据

```sql
-- 使用触发器维护冗余
CREATE TRIGGER update_user_stats
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    UPDATE user_profile_wide
    SET order_count = order_count + 1,
        total_spent = total_spent + NEW.amount,
        last_order_date = NEW.order_date
    WHERE user_id = NEW.user_id;
END;
```

## 实际设计建议

### OLTP系统（在线事务处理）
- 推荐范式：3NF或BCNF
- 重点：数据完整性、减少冗余
- 允许适度的反范式化用于关键查询

### OLAP系统（在线分析处理）
- 推荐范式：星型/雪花模型（维度建模）
- 重点：查询性能、分析便利性
- 大量采用反范式化

### 互联网应用
- 推荐：根据具体场景灵活选择
- 读写分离设计
- 缓存 + 反范式化优化热点查询

## 面试要点

### 常见问题

**Q1: 什么是数据库范式？**
> 范式是关系数据库设计的规范，用于减少数据冗余、避免更新异常。主要范式包括1NF（原子性）、2NF（消除部分依赖）、3NF（消除传递依赖）、BCNF等。

**Q2: 2NF和3NF的区别？**
> 2NF消除非主属性对主键的部分依赖（针对复合主键）；3NF消除非主属性对主键的传递依赖。3NF在2NF基础上进一步减少冗余。

**Q3: 什么时候需要反范式化？**
> 当查询性能成为瓶颈、读操作远多于写操作、需要实时响应复杂查询时，可以适当反范式化。但需要额外机制维护数据一致性。

**Q4: BCNF和3NF的关系？**
> BCNF是3NF的加强版。3NF允许主属性依赖于非主属性，BCNF不允许任何属性依赖于非码属性。所有BCNF都满足3NF，但反之不成立。
