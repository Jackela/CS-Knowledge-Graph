# CAP定理 (CAP Theorem)

## 简介

**CAP定理**（又称**布鲁尔定理, Brewer's Theorem**）是分布式系统领域的核心理论，由Eric Brewer于2000年提出。它指出：在一个分布式系统中，**一致性（Consistency）**、**可用性（Availability）**、**分区容错性（Partition Tolerance）**这三个属性**不可能同时满足**，最多只能同时满足其中两个。

```
CAP三角:

       一致性 (Consistency)
            /\
           /  \
          /    \
         /      \
        /   CAP   \
       /    定理    \
      /______________\
分区容错性          可用性
(Partition          (Availability)
 Tolerance)
```

## 三个核心属性

### 1. 一致性 (Consistency)

**定义**：所有节点在同一时间看到的数据是一致的。读操作总能返回最新的写操作结果。

**特点**：
- 强一致性：每次读取都返回最新写入的值
- 弱一致性：可能读取到旧值，但最终会一致
- 最终一致性：不保证立即一致，但保证最终会一致

```
一致性的要求:

客户端A          节点1          节点2          客户端B
  │    写入x=1    │              │              │
  │──────────────>│              │              │
  │               │   同步x=1    │              │
  │               │─────────────>│              │
  │               │   确认OK     │              │
  │    写入成功   │<─────────────│              │
  │<──────────────│              │              │
  │                              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回x=1    │
  │                              │─────────────>│
  │                              │              │
  必须读取到最新值x=1
```

### 2. 可用性 (Availability)

**定义**：每个请求都能在合理的时间内获得非错误的响应，但不保证返回的是最新数据。

**特点**：
- 系统总是可用，不返回错误
- 不保证数据是最新的
- 部分节点故障时仍能提供服务

```
可用性的要求:

客户端A          节点1(故障)      节点2          客户端B
  │    写入x=1    │     X        │              │
  │──────────────>│    故障      │              │
  │               │              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回x=0    │
  │                              │─────────────>│
  │                              │              │
  即使节点1故障，节点2仍能响应
  (可能返回旧值x=0，但不是错误)
```

### 3. 分区容错性 (Partition Tolerance)

**定义**：系统在遇到网络分区（节点间通信中断）时仍能继续运行。

**特点**：
- 网络分区不可避免
- 分布式系统必须容忍分区
- 分区时需要在C和A之间做选择

```
网络分区场景:

    节点A  <--❌网络中断❌-->  节点B
      │                           │
      │ 分区1                      │ 分区2
      │                           │
    客户端1                       客户端2
      
分区发生后:
- 如果保证一致性(C)：必须拒绝写入，牺牲可用性(A)
- 如果保证可用性(A)：允许各自写入，牺牲一致性(C)
```

## CAP组合与系统类型

### CP系统 (Consistency + Partition Tolerance)

**牺牲可用性，保证一致性和分区容错**

| 系统 | 特点 |
|------|------|
| **etcd** | 强一致性，用于配置管理 |
| **ZooKeeper** | 协调服务，强一致性 |
| **Consul** | 服务发现，CP模式 |
| **MongoDB (配置为CP)** | 主从复制，可配置 |
| **HBase** | 基于HDFS，强一致性 |

**适用场景**：
- 金融交易系统
- 库存管理
- 配置中心
- 对一致性要求极高的场景

```
CP系统行为:

客户端A          主节点          从节点(分区)      客户端B
  │    写入x=1    │              │     X        │
  │──────────────>│              │   分区       │
  │               │   同步失败   │              │
  │               │────❌───────>│              │
  │    写入失败   │              │              │
  │<──────────────│              │              │
  │   (拒绝写入)   │              │              │
  │                              │    读取x     │
  │                              │<─────────────│
  │                              │   返回旧值   │
  │                              │   或超时     │
```

### AP系统 (Availability + Partition Tolerance)

**牺牲一致性，保证可用性和分区容错**

| 系统 | 特点 |
|------|------|
| **Cassandra** | 最终一致性，高可用 |
| **DynamoDB** | 亚马逊KV存储，AP模式 |
| **Couchbase** | 分布式文档数据库 |
| **Riak** | 分布式KV存储 |
| **Eureka** | 服务发现，AP模式 |

**适用场景**：
- 社交网络
- 内容分发
- 实时分析
- 对可用性要求极高的场景

```
AP系统行为:

客户端A          节点A(分区1)    节点B(分区2)      客户端B
  │    写入x=1    │              │              │
  │──────────────>│              │              │
  │    写入成功   │              │              │
  │<──────────────│              │              │
  │    (继续服务)  │     X        │              │
  │               │   网络分区    │              │
  │                              │    写入x=2    │
  │                              │<─────────────│
  │                              │    写入成功   │
  │                              │─────────────>│
  │                              │              │
  恢复后需要解决冲突:x=1 vs x=2
```

### CA系统 (Consistency + Availability)

**理论上存在，实际分布式系统中不可能**

- 单机系统可以实现CA
- 分布式系统必须有P（分区容错）
- 因为网络分区不可避免

```
CA系统的局限:

┌─────────────────────────────────────┐
│  单机数据库 (如MySQL单节点)          │
│  - 一致性: ✓ 没有复制延迟           │
│  - 可用性: ✓ 单机正常即可用         │
│  - 分区容错: ✗ 单机无网络问题       │
└─────────────────────────────────────┘
         ↓
    不是真正的分布式系统
```

## 实际系统中的CAP权衡

### 1. 关系型数据库 (RDBMS)

**单节点**: CA
- 一致性：✓ ACID保证
- 可用性：✓ 单机正常即可用
- 分区容错：✗ 单节点

**主从复制**: 可配置为CP或AP
- 同步复制 → CP
- 异步复制 → AP（最终一致性）

### 2. NoSQL数据库

| 数据库 | 类型 | CAP选择 | 说明 |
|--------|------|---------|------|
| **MongoDB** | 文档 | CP/AP可选 | 主从可配置 |
| **Cassandra** | 列族 | AP | 可调一致性级别 |
| **Redis** | KV | CP | 主从同步 |
| **CouchDB** | 文档 | AP | 多主复制 |

### 3. 分布式协调服务

| 系统 | CAP选择 | 用途 |
|------|---------|------|
| **ZooKeeper** | CP | 分布式协调 |
| **etcd** | CP | Kubernetes配置 |
| **Consul** | CP/AP可选 | 服务发现 |

## PACELC定理 (CAP的扩展)

**PACELC**进一步细化了CAP：

> 如果有分区(**P**)，系统必须在可用性(**A**)和一致性(**C**)间选择；
> 否则(**E**lse)，即使没有分区，也必须在延迟(**L**)和一致性(**C**)间选择。

```
PACELC分析:

P(分区发生)
  │
  ├─ 选择A ──> 牺牲一致性，保证可用性
  │
  └─ 选择C ──> 牺牲可用性，保证一致性

E(Else - 无分区)
  │
  ├─ 选择L ──> 低延迟，可能读到旧数据
  │
  └─ 选择C ──> 强一致性，可能高延迟
```

## 面试要点

### Q1: 为什么分布式系统必须有P？

**答案**：
1. 网络分区在分布式系统中**不可避免**
2. 网络延迟、丢包、节点故障都会导致分区
3. 如果不能容忍分区，系统就退化为单机
4. 因此分布式系统实际上是在**C和A之间选择**

### Q2: ZooKeeper是CP还是AP？

**答案**：
- **ZooKeeper是CP系统**
- 使用ZAB协议保证强一致性
- 当大多数节点可用时才能写入
- 牺牲部分可用性换取一致性

### Q3: 如何选择CP还是AP？

**决策流程**：

```
                    业务场景
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
    金融交易       社交网络        配置中心
    库存扣减       点赞计数        服务发现
        │             │             │
        ▼             ▼             ▼
    必须CP          可以AP          通常CP
    (不能超卖)      (可以不一致)     (必须一致)
        │             │             │
        └─────────────┴─────────────┘
                      │
              选择相应的系统
```

### Q4: CAP与ACID的关系？

| ACID | 对应CAP | 说明 |
|------|---------|------|
| **原子性(Atomicity)** | C | 事务完整性 |
| **一致性(Consistency)** | C | 数据正确性 |
| **隔离性(Isolation)** | C | 并发控制 |
| **持久性(Durability)** | A | 数据不丢失 |

**关系**：ACID是数据库事务特性，CAP是分布式系统理论。ACID的一致性是数据完整性约束，CAP的一致性是副本一致性。
