# 分片 (Sharding)

## 简介

**分片 (Sharding)** 是将大型数据库或数据集分割成更小、更易管理的部分（分片）的技术。分片是数据库水平扩展的核心手段，用于解决单节点性能瓶颈和存储限制。

```
┌─────────────────────────────────────────────────────────────┐
│                  分片 vs 不分片对比                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  不分片（垂直扩展）              分片（水平扩展）             │
│                                                             │
│  ┌─────────────────┐           ┌─────────┐ ┌─────────┐     │
│  │                 │           │ Shard 1 │ │ Shard 2 │     │
│  │   大型数据库     │           │ 0-500万 │ │500-1000万│    │
│  │   1000万用户     │    =>     │ 用户A-M │ │ 用户N-Z │     │
│  │   单点瓶颈       │           ├─────────┤ ├─────────┤     │
│  │   容量受限       │           │ Shard 3 │ │ Shard 4 │     │
│  │                 │           │1000-1500万│1500-2000万│    │
│  └─────────────────┘           └─────────┘ └─────────┘     │
│                                                             │
│  成本指数增长                   成本线性增长                 │
│  有上限                         理论无限                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 分片策略

### 1. 哈希分片 (Hash Sharding)

```
哈希分片原理：

shard = hash(key) % shard_count

示例：用户ID分片
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  user_id = "user_12345"                                     │
│  hash(user_id) = 987654321                                  │
│  shard = 987654321 % 4 = 1                                  │
│                                                             │
│  分片分布：                                                  │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐           │
│  │ Shard 0 │ │ Shard 1 │ │ Shard 2 │ │ Shard 3 │           │
│  │ hash%4=0│ │ hash%4=1│ │ hash%4=2│ │ hash%4=3│           │
│  └─────────┘ └────▲────┘ └─────────┘ └─────────┘           │
│                   │                                         │
│              user_12345 存储位置                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**优点：**
- 数据分布均匀
- 避免热点问题

**缺点：**
- 范围查询需要访问所有分片
- 扩容时需要重新分配数据（rehashing）

```python
# 一致性哈希（解决扩容问题）
import hashlib

class ConsistentHashRing:
    def __init__(self, replicas=3):
        self.replicas = replicas
        self.ring = {}
        self.sorted_keys = []
    
    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)
    
    def add_node(self, node):
        for i in range(self.replicas):
            key = self._hash(f"{node}:{i}")
            self.ring[key] = node
            self.sorted_keys.append(key)
        self.sorted_keys.sort()
    
    def remove_node(self, node):
        for i in range(self.replicas):
            key = self._hash(f"{node}:{i}")
            del self.ring[key]
            self.sorted_keys.remove(key)
    
    def get_node(self, key):
        if not self.ring:
            return None
        hash_key = self._hash(key)
        for sorted_key in self.sorted_keys:
            if hash_key <= sorted_key:
                return self.ring[sorted_key]
        return self.ring[self.sorted_keys[0]]
```

### 2. 范围分片 (Range Sharding)

```
范围分片原理：

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  按时间范围分片：                                            │
│                                                             │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐   │
│  │ 2023-Q1   │ │ 2023-Q2   │ │ 2023-Q3   │ │ 2023-Q4   │   │
│  │Shard 1    │ │Shard 2    │ │Shard 3    │ │Shard 4    │   │
│  │Jan-Mar   │ │Apr-Jun   │ │Jul-Sep   │ │Oct-Dec   │   │
│  └───────────┘ └───────────┘ └───────────┘ └───────────┘   │
│                                                             │
│  按ID范围分片：                                              │
│                                                             │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐   │
│  │ 1-1M      │ │ 1M-2M     │ │ 2M-3M     │ │ 3M-4M     │   │
│  │Shard 1    │ │Shard 2    │ │Shard 3    │ │Shard 4    │   │
│  └───────────┘ └───────────┘ └───────────┘ └───────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**优点：**
- 范围查询高效
- 扩容方便（只需添加新分片）

**缺点：**
- 可能存在热点（如最新数据集中在最后一个分片）
- 需要合理设计分片键范围

### 4. 复合分片 (Composite Sharding)

```
组合多种策略：

第一层：按地区列表分片
    ├── 中国区
    │     └── 第二层：按用户ID哈希分片
    │           ├── Shard-CN-1
    │           ├── Shard-CN-2
    │           └── Shard-CN-3
    ├── 美国区
    │     └── 按用户ID哈希分片
    └── 欧洲区
          └── 按用户ID哈希分片
```

## 分片键选择

### 选择原则

```
┌─────────────────────────────────────────────────────────────┐
│                   分片键选择原则                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 基数 Cardinality                                        │
│     └── 选择高基数字段（取值范围广）                          │
│     └── 避免低基数字段（如性别：男/女）                        │
│                                                             │
│  2. 频率 Frequency                                          │
│     └── 避免热点字段                                         │
│     └── 如时间戳会导致最新分片成为热点                        │
│                                                             │
│  3. 单调性 Monotonicity                                     │
│     └── 避免单调递增字段（如自增ID）                          │
│     └── 会导致数据集中在最后一个分片                          │
│                                                             │
│  4. 查询模式 Query Pattern                                  │
│     └── 选择与查询条件匹配的字段                              │
│     └── 常用查询字段作为分片键可减少跨分片查询                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 常见分片键示例

| 场景 | 推荐分片键 | 策略 | 说明 |
|------|-----------|------|------|
| 用户系统 | user_id | 哈希 | 均匀分布，单用户查询高效 |
| 订单系统 | (user_id, 时间) | 复合 | 支持用户维度和时间维度查询 |
| 日志系统 | timestamp | 范围 | 支持时间范围查询，易于归档 |
| 地理位置 | geo_hash | 哈希 | 支持地理邻近查询 |
| 多租户 | tenant_id | 列表 | 数据隔离，便于管理 |

## 跨分片操作

### 跨分片查询

```
跨分片聚合查询：

SELECT COUNT(*) FROM orders WHERE status = 'completed'

执行过程：
┌──────────┐
│  查询    │
└────┬─────┘
     │
     ├──▶ Shard 1: COUNT = 1000 ──┐
     │                             │
     ├──▶ Shard 2: COUNT = 1500 ──┼──▶ 汇总 = 4500
     │                             │
     └──▶ Shard 3: COUNT = 2000 ──┘

优化：
- 使用聚合管道（如MapReduce）
- 建立全局汇总表
- 使用数据仓库进行离线分析
```

### 跨分片事务

```
跨分片事务处理：

┌──────────┐              ┌──────────┐              ┌──────────┐
│ 协调器   │◀────────────▶│  Shard 1 │◀────────────▶│  Shard 2 │
└────┬─────┘              └──────────┘              └──────────┘
     │
     │ 1. 准备阶段
     ├────────────────────▶Prepare
     ├────────────────────────────────────────────────▶Prepare
     │
     │ 2. 提交阶段
     ├────────────────────▶Commit
     ├────────────────────────────────────────────────▶Commit
     │
     ▼
  完成

挑战：
- 性能开销大
- 死锁风险
- 推荐避免跨分片事务
```

## 扩容与缩容

### 平滑扩容

```
扩容过程：

阶段1：准备
┌──────────┐ ┌──────────┐ ┌──────────┐
│ Shard 1  │ │ Shard 2  │ │ [新Shard]│  ← 添加新分片
│(0-500万) │ │(500-1000万)│ │(1000万+ )│
└──────────┘ └──────────┘ └──────────┘

阶段2：双写
应用 ──▶ 写入旧分片
    ──▶ 同时写入新分片（异步）

阶段3：数据迁移
使用后台任务将部分数据从旧分片迁移到新分片

阶段4：切换
更新路由规则，将新数据路由到新分片

阶段5：清理
停止双写，删除旧分片中已迁移的数据
```

### 一致性哈希扩容

```
一致性哈希的优势：

扩容前：                    扩容后（添加Shard 4）：

    Shard 1                   Shard 1
      ●                         ●
    /   \                     /   \
   /     \                   /  ●  \  ← Shard 4
  ●───────●                 ●───────●
Shard 2   Shard 3         Shard 2   Shard 3

只有Shard 1和Shard 2之间的数据需要迁移到Shard 4
大部分数据不需要移动（约1/N的数据需要迁移）
```

## 实践案例

### 用户表分片

```sql
-- 用户表分片方案
-- 分片键：user_id（哈希分片）
-- 分片数：16

-- 创建分片表（MySQL + ShardingSphere）
CREATE TABLE user_0 LIKE user_template;
CREATE TABLE user_1 LIKE user_template;
...
CREATE TABLE user_15 LIKE user_template;

-- ShardingSphere配置
sharding:
  tables:
    user:
      actualDataNodes: ds_${0..3}.user_${0..15}
      tableStrategy:
        standard:
          shardingColumn: user_id
          shardingAlgorithmName: user_hash
      databaseStrategy:
        standard:
          shardingColumn: user_id
          shardingAlgorithmName: db_hash
  
  shardingAlgorithms:
    user_hash:
      type: INLINE
      props:
        algorithm-expression: user_${user_id % 16}
    db_hash:
      type: INLINE
      props:
        algorithm-expression: ds_${user_id % 4}
```

### 订单表分片

```sql
-- 订单表分片方案
-- 分片键：(user_id, create_time) 复合分片

-- 按用户ID哈希分库，按时间范围分表
CREATE TABLE order_202401 LIKE order_template;
CREATE TABLE order_202402 LIKE order_template;
...

-- 双维度查询优化
-- 1. 按用户查询：直接路由到对应分片
SELECT * FROM order WHERE user_id = 12345;

-- 2. 按时间查询：广播到所有分片，汇总结果
SELECT * FROM order WHERE create_time > '2024-01-01';
```

## 常见问题

### 热点问题

```
热点产生：

时间分片热点：            用户分片热点：
                          
Shard N-1 ████            Shard K ████████████  ← 大V用户
Shard N   ████████████    Others  ██  ██  ██
          ↑                      
    最新数据集中          数据分布不均

解决方案：
1. 时间分片 + 哈希：timestamp + hash(user_id)
2. 热点数据单独处理：大V用户单独分片
3. 缓存热点数据
```

### 数据倾斜

```
数据倾斜检测：

Shard 1: 100万条（10%）
Shard 2: 150万条（15%）
Shard 3: 800万条（65%）  ← 严重倾斜
Shard 4: 100万条（10%）

解决方案：
1. 重新选择分片键
2. 使用一致性哈希
3. 手动迁移数据
4. 虚拟节点技术
```

## 面试要点

### 常见问题

**Q1: 分片和分区的区别？**
> 分区是数据库内部将大表逻辑分割，数据仍在同一节点；分片是将数据分布到不同节点。分区是垂直切分，分片是水平切分。

**Q2: 如何选择分片键？**> 选择高基数、查询频繁的字段，避免单调递增字段，考虑数据分布均匀性和查询模式。

**Q3: 一致性哈希的原理和优点？**> 一致性哈希将节点和数据映射到环形空间，添加/删除节点只影响相邻节点的数据。优点是扩容时只需迁移少量数据（约1/N），避免全量rehash。

**Q4: 跨分片查询如何优化？**> 1. 尽量避免跨分片查询
> 2. 使用分片键进行路由
> 3. 建立全局汇总表
> 4. 使用数据仓库离线分析
> 5. 并行查询各分片汇总结果
