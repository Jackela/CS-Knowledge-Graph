# 需求工程 (Requirements Engineering)

## 概念

需求工程（Requirements Engineering）是软件工程中系统化管理软件需求的学科，涵盖需求的获取、分析、规格说明、验证和变更管理。

## 需求类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **功能需求** | 系统应提供的功能 | 用户可以发布评论 |
| **非功能需求** | 质量属性和约束 | 系统响应时间 < 2秒 |
| **业务需求** | 高层业务目标 | 提升用户留存率20% |
| **用户需求** | 用户的具体期望 | 支持深色模式 |

## 需求获取技术

- **访谈**: 与利益相关者一对一交流
- **问卷**: 大规模收集用户意见
- **观察**: 实地观察用户工作流程
- **原型**: 通过可交互原型收集反馈
- **头脑风暴**: 团队协作产生创意
- **文档分析**: 研究现有系统和流程文档

## 需求分析方法

### 用例分析
```
用例名称: 用户登录
参与者: 注册用户
前置条件: 用户已注册
基本流程:
  1. 用户输入用户名和密码
  2. 系统验证凭据
  3. 系统授予访问权限
后置条件: 用户进入系统主页
```

### 用户故事
```
作为 [角色]
我想要 [功能]
以便 [价值]

示例:
作为 在线购物者
我想要 保存商品到收藏夹
以便 稍后购买
```

## 需求规格说明

### SRS文档结构

1. **引言**: 目的、范围、定义
2. **总体描述**: 产品视角、用户特征
3. **系统特性**: 详细功能需求
4. **外部接口**: 用户、硬件、软件接口
5. **非功能需求**: 性能、安全、可用性
6. **其他需求**: 数据库、国际化等

## 需求验证

- **评审**: 同行审查需求文档
- **原型验证**: 通过原型确认需求理解
- **模型验证**: 使用形式化方法验证
- **测试用例生成**: 确保需求可测试

## 需求变更管理

### 变更控制流程

```
变更请求 → 影响分析 → 决策(批准/拒绝)
                ↓
         实施变更 → 验证 → 更新基线
```

### 常见挑战

- **需求蔓延**: 不受控制的范围扩大
- **镀金**: 添加不必要功能
- **遗漏**: 关键需求未识别
- **冲突**: 需求之间相互矛盾
