# 测试驱动开发 (TDD - Test Driven Development)

## 概念

测试驱动开发（TDD）是一种**软件开发方法论**，要求在编写实际代码之前先编写测试用例。

> **核心思想**: 以测试为驱动，小步快跑，持续重构。

## TDD循环（红-绿-重构）

```
┌─────────────────────────────────────┐
│           TDD 循环                   │
├─────────────────────────────────────┤
│                                     │
│   ┌──────────┐                     │
│   │  1. 编写  │  写一个失败的测试    │
│   │  失败测试 │  (Red)              │
│   └─────┬────┘                     │
│         │                          │
│         ▼                          │
│   ┌──────────┐                     │
│   │  2. 编写  │  写最少代码使测试通过│
│   │  通过代码 │  (Green)            │
│   └─────┬────┘                     │
│         │                          │
│         ▼                          │
│   ┌──────────┐                     │
│   │  3. 重构  │  优化代码结构       │
│   │          │  (Refactor)         │
│   └─────┬────┘                     │
│         │                          │
│         └──────────▶ 回到步骤1     │
│                                     │
└─────────────────────────────────────┘
```

## 三条规则

1. **不允许编写任何产品代码，除非它能让失败的单元测试通过**
2. **不允许编写过多的单元测试，足以导致失败即可**
3. **不允许编写多于刚好能让测试通过的产品代码**

## TDD的优势

| 优势 | 说明 |
|------|------|
| 设计驱动 | 测试先行促使更好的API设计 |
| 安全网 | 测试覆盖率保证重构安全 |
| 需求澄清 | 测试即需求，明确预期行为 |
| 调试减少 | 问题在早期发现并解决 |
