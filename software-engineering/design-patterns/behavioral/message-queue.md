# 消息队列 (Message Queue)

消息队列（Message Queue）是一种**进程间通信或异步通信**的方式，用于在分布式系统中存储和转发消息。它实现了生产者和消费者的解耦，是构建可扩展、可靠系统的关键组件。

## 核心概念

### 消息队列模型

```
点对点模型 (Point-to-Point)：          发布/订阅模型 (Pub/Sub)：

┌──────────┐                         ┌──────────┐
│ Producer │                         │ Producer │
└────┬─────┘                         └────┬─────┘
     │                                    │
     ▼                                    ▼
┌──────────┐                         ┌──────────┐
│  Queue   │                         │  Topic   │
│  (队列)   │                         │  (主题)   │
└────┬─────┘                         └────┬─────┘
     │                                    │
     ▼                                    ▼
┌──────────┐                    ┌─────────┬─────────┐
│ Consumer │                    │ Consumer│ Consumer│
└──────────┘                    └─────────┴─────────┘

特点：                              特点：
- 一条消息只被一个消费者消费        - 一条消息可被多个消费者消费
- 支持负载均衡                      - 支持广播
- 消息消费后从队列移除              - 消费者订阅特定主题
```

### 消息队列核心功能

```
┌─────────────────────────────────────────────────────────────┐
│                  消息队列核心功能                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 解耦 (Decoupling)                                        │
│     生产者和消费者独立运行，互不影响                         │
│                                                             │
│  2. 异步 (Asynchronous)                                      │
│     生产者发送消息后立即返回，无需等待处理                   │
│                                                             │
│  3. 削峰填谷 (Peak Shaving)                                  │
│     高并发时缓冲消息，平滑处理峰值                           │
│                                                             │
│  4. 可靠投递 (Reliable Delivery)                             │
│     消息持久化，确保不丢失                                   │
│                                                             │
│  5. 顺序保证 (Ordering)                                      │
│     支持消息的有序消费                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 主流消息队列

### 对比概览

| 特性 | RabbitMQ | Kafka | RocketMQ | Pulsar |
|------|----------|-------|----------|--------|
| **协议** | AMQP | 自定义 | 自定义 | 自定义 |
| **架构** | 主从复制 | 分布式 | 分布式 | 分层存储 |
| **吞吐量** | 万级/秒 | 百万级/秒 | 十万级/秒 | 百万级/秒 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 | 毫秒级 |
| **消息持久化** | ✅ | ✅ | ✅ | ✅ |
| **消息回溯** | ❌ | ✅ | ✅ | ✅ |
| **多租户** | ❌ | ❌ | ✅ | ✅ |
| **适用场景** | 通用消息 | 日志/流处理 | 金融/电商 | 云原生 |

### RabbitMQ

```
RabbitMQ 特点：
- 基于 Erlang/OTP，稳定性高
- 支持多种消息模式（direct, topic, fanout, headers）
- 灵活的路由规则
- 丰富的插件生态

使用场景：
- 企业级消息通信
- 复杂路由需求
- 延迟队列
- 优先级队列
```

### Kafka

```
Kafka 特点：
- 高吞吐量，低延迟
- 消息持久化到磁盘，支持回溯
- 分区和副本机制
- 与流处理集成（Kafka Streams）

架构：
Producer ──▶ Topic/Partition ──▶ Consumer Group
                     │
                     ▼
              Kafka Brokers (Leader + Replicas)

使用场景：
- 日志收集
- 实时流处理
- 事件溯源
- 大数据管道
```

## 消息可靠性

### 消息投递语义

| 语义 | 描述 | 实现方式 |
|------|------|----------|
| **At most once** | 最多一次，可能丢失 | 异步发送，失败不重试 |
| **At least once** | 至少一次，可能重复 | 发送失败重试，消费幂等 |
| **Exactly once** | 恰好一次 | 事务 + 幂等生产者 |

### 可靠性保证

```python
# Kafka 可靠性配置示例
producer_config = {
    # 确认机制
    'acks': 'all',              # 等待所有副本确认
    
    # 重试配置
    'retries': 3,               # 发送失败重试次数
    'retry.backoff.ms': 1000,   # 重试间隔
    
    # 幂等性
    'enable.idempotence': True,  # 启用幂等生产者
    
    # 事务（需要 enable.idempotence=True）
    'transactional.id': 'my-producer-id'
}

consumer_config = {
    # 自动提交偏移量（默认）
    'enable.auto.commit': False,  # 手动提交更安全
    
    # 消费位置
    'auto.offset.reset': 'earliest',  # 从最早开始消费
    
    # 消费者组
    'group.id': 'my-consumer-group'
}
```

## 消息顺序

### 顺序保证策略

```
全局顺序 vs 分区顺序：

全局顺序（单分区）：                  分区顺序（多分区）：
┌─────────────────┐                  ┌─────┐ ┌─────┐ ┌─────┐
│  Partition 0    │                  │ P0  │ │ P1  │ │ P2  │
│  ─────────────  │                  │─────│ │─────│ │─────│
│  Msg1 → Msg2    │                  │ M1  │ │ M4  │ │ M7  │
│  → Msg3 → Msg4  │                  │ M2  │ │ M5  │ │ M8  │
└─────────────────┘                  │ M3  │ │ M6  │ │ M9  │
                                     └─────┘ └─────┘ └─────┘
                                     
全局有序但吞吐受限                    分区有序，吞吐更高
                                      相同 Key 的消息在同一个分区
```

```python
# Kafka 消息顺序保证
# 方式1：单分区（全局有序）
producer.send('my-topic', value=data)  # 不指定 key，轮询分区

# 方式2：按 Key 分区（分区有序）
producer.send('my-topic', key=user_id, value=data)
# 相同 user_id 的消息会发送到同一个分区
```

### 1. 异步处理

```
订单处理流程：

同步处理：                          异步处理：
┌──────────┐                       ┌──────────┐
│ 用户下单  │                       │ 用户下单  │
└────┬─────┘                       └────┬─────┘
     │                                  │
     ▼                                  ▼
┌──────────┐                       ┌──────────┐
│ 创建订单  │  100ms                │ 创建订单  │  50ms
└────┬─────┘                       └────┬─────┘
     │                                  │
     ▼                                  ▼
┌──────────┐                       ┌──────────┐
│ 扣减库存  │  200ms                │ 返回结果  │
└────┬─────┘                       └──────────┘
     │                                  │
     ▼                                  │ 异步处理：
┌──────────┐                           │ 扣减库存
│ 发送通知  │  100ms                    │ 发送通知
└────┬─────┘                           │ 积分计算
     │                                  ▼
     ▼                              ┌──────────┐
┌──────────┐                       │ 消息队列  │
│ 积分计算  │  150ms                └──────────┘
└────┬─────┘
     │
     ▼
返回结果  总耗时：550ms

异步总耗时：50ms（用户感知）
```
