# 技术债务 (Technical Debt)

## 概念

技术债务（Technical Debt）是指为了快速交付而采取的权宜之计，导致代码或设计质量下降，未来需要额外工作来修复。

> **核心思想**: 短期捷径带来的长期成本。

## 技术债务的类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **故意债务** | 明知故犯，为了赶进度 | "先上线，再重构" |
| **无意债务** | 由于缺乏知识导致的 | 不懂设计模式 |
| **腐烂债务** | 代码逐渐老化 | 长期不维护的代码 |
| **环境债务** | 外部依赖过时 | 使用EOL的技术 |

## 技术债务的四象限

```
                 故意
                   │
    轻率 ──────────┼────────── 谨慎
                   │
    "没时间设计"    │    "必须今天发布"
                   │
    ───────────────┼───────────────
                   │
    "不知道 better" │    "现在知道 better"
                   │
    轻率 ──────────┼────────── 谨慎
                   │
                 无意
```

## 管理技术债务

### 识别债务
- 代码异味扫描
- 架构审查
- 团队反馈

### 量化债务
- 代码质量指标
- 缺陷率统计
- 开发速度下降

### 偿还策略
1. **持续重构** - 边开发边清理
2. **预留时间** - Sprint中分配重构时间
3. **专项重构** - 集中处理重大债务
