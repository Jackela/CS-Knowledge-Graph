# 分支策略 (Branching Strategy)

分支策略是团队协作中管理代码分支的规范和流程，确保代码质量并支持并行开发。

## 常见策略

### 1. Git Flow

```
┌─────────────────────────────────────────────────────────────┐
│                       Git Flow                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  master/main                                                 │
│  ●────────────────────────●────────────────●────────────▶   │
│  │                        │                │                 │
│  │  hotfix-1.1      ●─────┴────●           │                 │
│  │                  │          │           │                 │
│  develop            │          │           │                 │
│  ●──────────────────┼──────────┼───────────┼──────────────▶  │
│  │      ●───────────┘          │    ●──────┴────●            │
│  │      │                      │    │          │             │
│  │  feature-A                  │  feature-B                 │
│  │  ●───●                      │  ●──●          ●            │
│  │      │                      │  │             │             │
│  │      ●─────────────────────┼──┘             │             │
│  │                             │                │             │
│  release-1.0                  release-1.1                   │
│  │                             │                             │
│  ●─────────────────────────────┘                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**分支说明：**
| 分支 | 用途 | 来源 | 合并到 |
|------|------|------|--------|
| master | 生产环境 | - | - |
| develop | 开发集成 | master | master, release |
| feature/* | 功能开发 | develop | develop |
| release/* | 版本发布 | develop | master, develop |
| hotfix/* | 紧急修复 | master | master, develop |

**适用场景：**
- 需要严格版本控制的项目
- 有计划的发布周期
- 大型团队协作

### 2. GitHub Flow

```
┌─────────────────────────────────────────────────────────────┐
│                     GitHub Flow                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  main                                                        │
│  ●────────●────────●────────●────────●────────●────────▶   │
│  │        │        │        │        │        │             │
│  │   ●────┘        │   ●────┘        │   ●────┘             │
│  │   │             │   │             │   │                   │
│  │ feature-A       │ feature-B       │ feature-C             │
│  │                 │                 │                       │
│  │  PR → Review → Merge                │                       │
│  │                                     │                       │
│  │                                     │                       │
│  └── Deploy to Production ────────────┘                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**特点：**
- 只有一个长期分支：main
- 功能分支从 main 创建
- 通过 Pull Request 合并
- 合并后立即部署

**适用场景：**
- 持续部署的项目
- Web 应用开发
- 敏捷开发团队

### 3. GitLab Flow

```
┌─────────────────────────────────────────────────────────────┐
│                     GitLab Flow                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  production                                                  │
│  ●────────●────────●────────●────────●────────●────────▶   │
│           │                      │                          │
│  pre-prod ●──────────────────────┘                          │
│           │                                                 │
│  main     ●────────●────────●────────●────────●────────▶   │
│           │        │        │        │                      │
│           │   ●────┘        │   ●────┘                      │
│           │   │             │   │                           │
│           │ feature-A       │ feature-B                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**特点：**
- 环境分支（main → pre-prod → production）
- 功能分支从 main 创建
- 合并到环境分支即部署

**适用场景：**
- 多环境部署
- 需要预发布环境
- 运维和开发协作

### 4. Trunk-Based Development

```
┌─────────────────────────────────────────────────────────────┐
│                Trunk-Based Development                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  trunk/main                                                  │
│  ●──●──●──●──●──●──●──●──●──●──●──●──●──●──●──●──●──●──▶  │
│     │     │  │  │  │        │     │  │                      │
│     ●─────┘  ●──┘  ●────────┘     ●──┘                      │
│     │        │     │              │                         │
│   feature  feature feature      feature                     │
│   (短生命周期)                                                │
│                                                             │
│   特性开关 (Feature Flags):                                   │
│   if (featureFlags.enabled('new-payment')) {                │
│       // 新功能代码                                          │
│   }                                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**特点：**
- 所有开发在主干上进行
- 功能分支生命周期极短（< 1天）
- 使用特性开关控制功能发布

**适用场景：**
- 持续集成/持续部署
- 需要快速迭代的产品
- 自动化测试完善的项目

## 选择指南

| 策略 | 团队规模 | 发布频率 | 复杂度 |
|------|---------|----------|--------|
| Git Flow | 大型 | 周期性发布 | 高 |
| GitHub Flow | 中小型 | 持续部署 | 低 |
| GitLab Flow | 中型 | 多环境部署 | 中 |
| Trunk-Based | 任何规模 | 持续集成 | 低 |

## 最佳实践

### 分支命名规范
```
feature/user-authentication
bugfix/login-error-1234
hotfix/security-patch
docs/api-reference
refactor/database-layer
```

### 提交信息规范
```
type(scope): subject

body

footer

类型：
- feat: 新功能
- fix: 修复
- docs: 文档
- style: 格式
- refactor: 重构
- test: 测试
- chore: 构建/工具
```
