## 定义

**代码审查 (Code Review)** 是软件开发过程中，由同行对源代码进行系统性检查的活动，旨在发现缺陷、提升代码质量、促进知识共享。

---

## 审查类型

| 类型 | 描述 | 适用场景 |
|------|------|----------|
| **结对审查** (Pair Review) | 两人实时共同审查 | 复杂逻辑、关键模块 |
| **团队审查** (Team Review) | 多人会议形式审查 | 架构变更、重大重构 |
| **工具辅助审查** (Tool-Assisted) | 通过PR/MR异步审查 | 日常开发、快速迭代 |
| **代码走查** (Walkthrough) | 作者讲解，他人提问 | 新人培训、知识传递 |

## 审查清单 (Checklist)

### 功能性审查

```
□ 代码是否实现了需求？
□ 边界条件是否处理？
□ 错误处理是否完善？
□ 是否有明显的逻辑错误？
```

### 可读性审查

```
□ 命名是否清晰表达意图？
□ 函数是否单一职责？
□ 注释是否必要且准确？
□ 代码结构是否清晰？
```

### 可维护性审查

```
□ 是否遵循[设计模式](./design-patterns.md)？
□ 是否违反[SOLID原则](./solid-principles.md)？
□ 重复代码是否可以提取？
□ 依赖是否合理？
```

### 性能审查

```
□ 算法复杂度是否合理？参见[时间复杂度](../references/time-complexity.md)
□ 是否有明显的性能瓶颈？
□ 资源使用是否合理？
□ 缓存使用是否得当？
```

### 安全性审查

```
□ 是否防范[常见漏洞](../security/common-vulnerabilities.md)？
□ 输入是否验证？
□ 敏感信息是否泄露？
□ 权限检查是否完善？
```

## 审查原则

### 1. 尊重与建设性

```
❌ 不良评论：
"这代码写得太烂了，完全看不懂"

✅ 建设性评论：
"这里的逻辑比较复杂，建议提取成一个函数，
 命名为 `calculateDiscount()` 会更清晰"
```

### 2. 对事不对人

```
❌ 针对人：
"你怎么又用全局变量了？"

✅ 针对代码：
"这个全局变量可能导致并发问题，建议改为参数传递"
```

### 3. 优先级区分

| 优先级 | 类别 | 示例 |
|--------|------|------|
| **阻塞** (Blocking) | 功能缺陷、安全漏洞 | 未处理异常、SQL注入 |
| **建议** (Suggestion) | 可优化但不强制 | 变量命名、注释完善 |
| **可选** (Nit) | 个人偏好 | 空行数量、括号位置 |

## 审查流程

```
┌─────────────────────────────────────────────────────────────┐
│                      代码审查流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 作者准备                                                │
│     ├── 自测通过                                            │
│     ├── 代码格式化                                          │
│     └── 添加必要注释                                        │
│                   │                                         │
│                   ▼                                         │
│  2. 提交审查                                                │
│     ├── 创建[Pull Request](../CONTRIBUTING.md)              │
│     ├── 描述变更内容                                        │
│     └── 指定审查者                                          │
│                   │                                         │
│                   ▼                                         │
│  3. 审查者审查                                              │
│     ├── 理解需求背景                                        │
│     ├── 逐行阅读代码                                        │
│     └── 提出评论/问题                                       │
│                   │                                         │
│                   ▼                                         │
│  4. 作者响应                                                │
│     ├── 讨论澄清                                            │
│     ├── 修改代码                                            │
│     └── 回复评论                                            │
│                   │                                         │
│                   ▼                                         │
│  5. 审查完成                                                │
│     ├── 所有阻塞问题解决                                    │
│     ├── 审查者批准                                          │
│     └── 代码合并                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 审查效率

### 最佳实践

```
□ 单次审查代码量 < 400行
□ 审查速度 < 500行/小时
□ 响应时间 < 24小时
□ 审查者人数：1-3人
```

### 统计指标

| 指标 | 说明 | 健康范围 |
|------|------|----------|
| **审查覆盖率** | 被审查代码比例 | > 80% |
| **平均审查时间** | 从提交到批准 | < 24小时 |
| **缺陷发现率** | 每千行发现问题数 | 因项目而异 |
| **返工率** | 需要多次审查的比例 | < 20% |
