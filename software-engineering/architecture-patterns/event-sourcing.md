# 事件溯源 (Event Sourcing)

## 概念

事件溯源（Event Sourcing）是一种**架构模式**，将系统状态的变化记录为一系列不可变的事件，而不是只存储当前状态。

> **核心思想**: 存储变化的历史，而非仅存储最终结果。

## 传统模式 vs 事件溯源

### 传统CRUD
```
数据库状态：User { name: "张三", age: 25 }

更新后：    User { name: "张三", age: 26 }

# 历史信息丢失！不知道 age 曾经是 25
```

### 事件溯源
```
事件流：
1. UserCreated { name: "张三", age: 25 }
2. UserAgeChanged { from: 25, to: 26 }
3. UserNameChanged { from: "张三", to: "张小三" }

# 当前状态可以通过重放所有事件得到
# 完整历史被保留
```

## 核心组件

```
┌─────────────────────────────────────────────┐
│              事件溯源架构                    │
├─────────────────────────────────────────────┤
│                                             │
│  Command ──▶ Aggregate ──▶ Event            │
│              (领域对象)     (状态变化)       │
│                  │                          │
│                  ▼                          │
│           Event Store                       │
│           (事件存储)                         │
│                  │                          │
│     ┌────────────┼────────────┐             │
│     ▼            ▼            ▼             │
│  Projector   Snapshot    Event Bus          │
│  (投影重建)   (快照)      (事件广播)         │
│     │                        │              │
│     ▼                        ▼              │
│  Read Model              Subscribers        │
│  (查询模型)               (订阅者)           │
│                                             │
└─────────────────────────────────────────────┘
```

## 适用场景

| 场景 | 说明 |
|------|------|
| 审计需求 | 需要完整的历史变更记录 |
| 复杂业务 | 业务流程复杂，状态变化多 |
| 回溯需求 | 需要恢复到历史状态 |
| 分析需求 | 需要基于历史数据分析 |

## 优缺点

**优点**:
- 完整的历史记录
- 天然支持审计
- 便于调试和问题追踪
- 可以重建任意时刻状态

**缺点**:
- 系统复杂度高
- 事件版本管理困难
- 查询需要投影重建
- 存储空间需求大
