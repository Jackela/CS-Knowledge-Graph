## 概述

**敏捷开发 (Agile Development)** 是一种以人为核心、迭代、循序渐进的软件开发方法。它强调**适应变化**胜过遵循计划，**交付可用软件**胜过详尽的文档。

### 敏捷宣言 (Agile Manifesto)

2001年，17位软件专家在雪鸟城共同发表了敏捷宣言：

```
┌─────────────────────────────────────────────────────────────────┐
│                    敏捷宣言核心价值观                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   我们践行敏捷开发，通过实践和帮助他人实践，                     │
│   发现更好的软件开发方法。通过这项工作，我们相信：               │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                                                         │   │
│   │   个体和互动     高于     流程和工具                     │   │
│   │   (Individuals and interactions over processes and tools)│   │
│   │                                                         │   │
│   │   可用的软件     高于     详尽的文档                     │   │
│   │   (Working software over comprehensive documentation)    │   │
│   │                                                         │   │
│   │   客户合作       高于     合同谈判                       │   │
│   │   (Customer collaboration over contract negotiation)     │   │
│   │                                                         │   │
│   │   响应变化       高于     遵循计划                       │   │
│   │   (Responding to change over following a plan)           │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   也就是说，尽管右项有其价值，但我们更重视左项的价值。           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 敏捷十二原则

```
┌─────────────────────────────────────────────────────────────────┐
│                    敏捷十二原则                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 最高优先级是通过持续交付有价值的软件来满足客户               │
│                                                                 │
│  2. 欢迎需求变更，即使在开发后期。敏捷流程利用变更为竞争优势   │
│                                                                 │
│  3. 频繁交付可用软件，周期从几周到几个月，越短越好               │
│                                                                 │
│  4. 业务人员和开发人员必须在整个项目中每天一起工作               │
│                                                                 │
│  5. 围绕被激励的个体构建项目，给予所需支持和信任                 │
│                                                                 │
│  6. 面对面交谈是最有效的信息传递方式                             │
│                                                                 │
│  7. 可用软件是进度的首要度量标准                                 │
│                                                                 │
│  8. 敏捷流程提倡可持续开发，保持稳定节奏                         │
│                                                                 │
│  9. 持续关注技术卓越和良好设计增强敏捷性                         │
│                                                                 │
│ 10. 简洁，即尽量减少不必要工作的艺术，至关重要                   │
│                                                                 │
│ 11. 最好的架构、需求和设计出自自组织团队                         │
│                                                                 │
│ 12. 团队定期反思如何更有效，并相应调整行为                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Scrum** 是最流行的敏捷框架之一，通过固定周期的迭代(Sprint)交付增量价值。

## Kanban方法

**Kanban (看板)** 是一种可视化工作流管理方法，强调**持续交付**和**限制在制品(WIP)**。

### Kanban vs Scrum

```
┌─────────────────────────────────────────────────────────────────┐
│                    Kanban vs Scrum 对比                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   特性            │    Scrum           │    Kanban              │
│   ────────────────┼────────────────────┼────────────────────    │
│   节奏            │  固定Sprint (2-4周) │  持续流动              │
│   角色            │  PO, SM, 团队       │  无强制角色            │
│   变更            │  Sprint内不变更     │  任何时间可变更        │
│   度量            │  速度 (Velocity)    │  前置时间/周期时间     │
│   最佳场景        │  需要定期交付       │  需要快速响应          │
│   仪式            │  固定的5个事件      │  无强制仪式            │
│                                                                 │
│   选择建议:                                                      │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ • 新团队/新项目 → 从Scrum开始，建立节奏                  │   │
│   │ • 运维/支持工作 → Kanban，快速响应变化                   │   │
│   │ • 成熟团队 → ScrumBan (两者结合)                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 敏捷 vs 瀑布

```
┌─────────────────────────────────────────────────────────────────┐
│                    敏捷 vs 瀑布模型                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   瀑布模型 (Waterfall):                                          │
│   ┌────┐   ┌────┐   ┌────┐   ┌────┐   ┌────┐   ┌────┐          │
│   │需求 │─>│设计 │─>│开发 │─>│测试 │─>│部署 │─>│维护 │          │
│   └────┘   └────┘   └────┘   └────┘   └────┘   └────┘          │
│      │        │        │        │        │                      │
│      └────────┴────────┴────────┘        │                      │
│                   变更成本高              │                      │
│                                          │                      │
│   特点: 阶段清晰、文档齐全、适合需求明确的项目                    │
│                                                                 │
│   ─────────────────────────────────────────────────────────     │
│                                                                 │
│   敏捷模型 (Agile):                                              │
│   ┌─────────────────────────────────────────────────────┐       │
│   │                      迭代 1                          │       │
│   │   ┌─────────────────────────────────────────────┐   │       │
│   │   │ 需求 │─>│设计 │─>│开发 │─>│测试 │─>│交付 │   │       │
│   │   └─────────────────────────────────────────────┘   │       │
│   └─────────────────────────────────────────────────────┘       │
│   ┌─────────────────────────────────────────────────────┐       │
│   │                      迭代 2                          │       │
│   │   ┌─────────────────────────────────────────────┐   │       │
│   │   │ 需求 │─>│设计 │─>│开发 │─>│测试 │─>│交付 │   │       │
│   │   └─────────────────────────────────────────────┘   │       │
│   └─────────────────────────────────────────────────────┘       │
│   ┌─────────────────────────────────────────────────────┐       │
│   │                      迭代 N...                       │       │
│   └─────────────────────────────────────────────────────┘       │
│                                                                 │
│   特点: 拥抱变化、快速交付、适合需求不确定的项目                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 详细对比

| 维度 | 瀑布模型 | 敏捷开发 |
|------|----------|----------|
| **需求** | 前期固定，后期难改 | 持续演化，欢迎变更 |
| **交付** | 末期一次性交付 | 持续增量交付 |
| **反馈** | 周期长，风险大 | 频繁，及时调整 |
| **文档** | 详尽完整 | 刚好足够 |
| **团队** | 按职能划分 | 跨职能自组织 |
| **适合** | 需求明确的大型项目 | 创新型、探索型项目 |
| **风险** | 后期集中暴露 | 早期发现和应对 |

## 面试要点

### Q1: Scrum和Kanban的核心区别？

**答案**：Scrum是迭代框架，有固定Sprint周期和仪式；Kanban是流方法，持续交付无固定节奏。Scrum适合需要定期节奏的团队，Kanban适合需要快速响应变化的运维团队。Scrum有强制角色，Kanban更灵活。

### Q2: 什么是"完成的定义"(Definition of Done)？

**答案**：DoD是团队对"工作完成"的共同标准。通常包括：代码完成、单元测试通过、代码审查通过、集成测试通过、文档更新、PO验收等。DoD确保交付质量，避免"差不多就好"。

### Q3: Sprint中需求变更怎么处理？

**答案**：原则上Sprint开始后不新增需求。如果必须变更：1) 取消当前Sprint重新开始(极端情况)；2) 与PO协商，用新需求替换等量的未完成需求；3) 记录到下一Sprint。频繁变更说明Sprint Planning质量需要提升。

### Q4: 敏捷估算为什么要用故事点而不是人天？

**答案**：故事点是相对估算，考虑了复杂度、不确定性和工作量。它不绑定具体人员，更稳定。人天受个人效率影响大，且容易变成"承诺"而非"估算"。故事点有助于计算团队速度(Velocity)，做更可靠的发布预测。
