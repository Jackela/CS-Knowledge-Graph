## 概述

在产品开发过程中，**需求文档**是连接业务、产品与技术的桥梁。三种核心文档构成了完整的需求传递体系：

| 文档 | 全称 | 受众 | 核心目标 |
|------|------|------|----------|
| **BRD** | Business Requirements Document | 高层管理者、业务部门 | 论证项目价值与可行性 |
| **PRD** | Product Requirements Document | 产品经理、设计师、开发 | 定义产品功能与用户体验 |
| **SRS** | Software Requirements Specification | 开发团队、测试团队 | 明确技术实现细节 |

```
┌─────────────────────────────────────────────────────────────────┐
│                    需求文档演进金字塔                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                    ┌─────────────┐                              │
│                    │     BRD     │  ← 为什么做? (商业价值)       │
│                    │   商业需求   │                              │
│                    └──────┬──────┘                              │
│                           │ 审批通过                             │
│                           ▼                                     │
│                    ┌─────────────┐                              │
│                    │     PRD     │  ← 做什么? (产品功能)         │
│                    │   产品需求   │                              │
│                    └──────┬──────┘                              │
│                           │ 需求评审                             │
│                           ▼                                     │
│                    ┌─────────────┐                              │
│                    │     SRS     │  ← 怎么做? (技术实现)         │
│                    │  软件规格    │                              │
│                    └─────────────┘                              │
│                                                                 │
│   抽象程度: 高 ──────────────────────────────────────> 低        │
│   受众:     管理层 ────────────────────────────────> 工程师      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## BRD (Business Requirements Document)

### 什么是BRD

**商业需求文档 (BRD)** 是从业务角度阐述"为什么要做这个项目"的文档。它聚焦于**商业价值**，用于获取高层支持和资源投入。

### BRD核心内容

```
┌─────────────────────────────────────────────────────────────────┐
│                    BRD 标准结构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 项目背景 (Background)                                       │
│     └─ 市场机会、业务痛点、战略契合度                             │
│                                                                 │
│  2. 商业目标 (Business Objectives)                              │
│     └─ 收入目标、用户增长、效率提升、成本节约                     │
│                                                                 │
│  3. 目标用户 (Target Audience)                                  │
│     └─ 用户画像、市场规模、用户价值主张                           │
│                                                                 │
│  4. 竞争分析 (Competitive Analysis)                             │
│     └─ 竞品对比、差异化优势、市场定位                             │
│                                                                 │
│  5. 预期收益 (Expected ROI)                                     │
│     └─ 收入预测、成本估算、投资回报率                             │
│                                                                 │
│  6. 风险评估 (Risk Assessment)                                  │
│     └─ 市场风险、技术风险、合规风险                               │
│                                                                 │
│  7. 里程碑计划 (Milestones)                                     │
│     └─ 关键节点、资源需求、成功标准                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### BRD示例：在线教育平台

```markdown
# 智能题库系统 BRD

## 1. 项目背景
- 现有题库更新滞后，无法满足个性化学习需求
- AI技术成熟，可实现智能推荐与自适应练习

## 2. 商业目标
- 12个月内DAU提升40%
- 付费转化率从5%提升至8%
- 减少人工出题成本60%

## 3. 目标用户
- 主要：K12学生 (市场规模：5000万)
- 次要：教师用户 (题库共建)

## 4. 竞争分析
| 竞品 | 优势 | 劣势 | 我们的机会 |
|------|------|------|-----------|
| X题库 | 题量大 | 无智能推荐 | AI个性化 |
| Y练习 | UI好 | 贵 | 性价比 |

## 5. 预期收益
- 首年新增收入：2000万
- 开发成本：800万
- ROI：150%

## 6. 风险评估
- 内容合规风险：中 (需审核机制)
- 技术实现风险：低 (AI方案成熟)

## 7. 里程碑
- Q1: 需求确认与原型验证
- Q2: MVP上线 (核心功能)
- Q3: 全面推广
```

## PRD (Product Requirements Document)

### 什么是PRD

**产品需求文档 (PRD)** 详细描述"产品要做什么功能"，是产品经理的核心产出。PRD将商业目标转化为具体的产品功能和用户体验。

### PRD核心内容

```
┌─────────────────────────────────────────────────────────────────┐
│                    PRD 标准结构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 文档概述 (Overview)                                         │
│     ├─ 修订历史、文档目的、范围说明                               │
│     └─ 术语表、参考资料                                          │
│                                                                 │
│  2. 产品背景 (Context)                                          │
│     ├─ 用户痛点、场景描述                                        │
│     └─ 成功指标 (Success Metrics)                                │
│                                                                 │
│  3. 用户故事/用例 (User Stories)                                │
│     └─ 作为[用户], 我想[功能], 以便[价值]                         │
│                                                                 │
│  4. 功能需求 (Functional Requirements)                          │
│     ├─ 功能清单、优先级 (MoSCoW)                                 │
│     └─ 详细功能描述、流程图、原型图                              │
│                                                                 │
│  5. 非功能需求 (Non-functional Requirements)                    │
│     ├─ 性能要求、安全要求                                        │
│     └─ 兼容性、可访问性                                          │
│                                                                 │
│  6. 界面设计 (UI/UX)                                            │
│     └─ 原型图、交互说明、设计规范                                │
│                                                                 │
│  7. 数据分析 (Analytics)                                        │
│     └─ 埋点需求、数据看板                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### PRD示例：智能题库功能模块

```markdown
# 智能题库系统 PRD

## 1. 文档概述
- 版本: v1.2
- 日期: 2024-01-15
- 作者: 产品经理小王

## 2. 产品背景
用户反馈现有练习模式单一，希望获得个性化推荐。

## 3. 用户故事
| ID | 用户故事 | 优先级 |
|----|---------|--------|
| US-01 | 作为学生，我想获得基于错题的针对性练习，以便巩固薄弱知识点 | Must |
| US-02 | 作为学生，我想查看学习进度报告，以便了解自己的提升情况 | Must |
| US-03 | 作为教师，我想快速组卷，以便节省备课时间 | Should |

## 4. 功能需求

### 4.1 智能推荐引擎
- **FR-01**: 系统根据用户历史答题记录，计算知识点掌握度
- **FR-02**: 系统每日生成个性化练习计划 (20题/天)
- **FR-03**: 推荐算法需考虑：难度梯度、知识点覆盖、遗忘曲线

### 4.2 错题本功能
```
流程图:
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 答题错误 │───>│ 自动收录 │───>│ 分类归档 │───>│ 定期复习 │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
                                    │
                                    ▼
                              ┌─────────┐
                              │ 同类推荐 │
                              └─────────┘
```

## 5. 非功能需求
- **性能**: 推荐结果生成时间 < 2秒
- **并发**: 支持10万用户同时在线
- **可用性**: 系统可用性 > 99.9%

## 6. 界面设计
[原型图链接]
关键页面：首页、练习页、报告页、错题本

## 7. 数据分析
- 埋点：推荐点击率、练习完成率、留存率
- 看板：日活、功能使用分布、推荐准确率
```

## SRS (Software Requirements Specification)

### 什么是SRS

**软件需求规格说明书 (SRS)** 是从技术角度描述"如何实现"的文档。它将PRD中的功能需求转化为可开发、可测试的技术规格。

### SRS核心内容

```
┌─────────────────────────────────────────────────────────────────┐
│                    SRS 标准结构                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 引言 (Introduction)                                         │
│     ├─ 目的、范围、定义、参考资料                                │
│     └─ 项目约束、假设与依赖                                      │
│                                                                 │
│  2. 总体描述 (Overall Description)                              │
│     ├─ 产品视角、用户特征                                        │
│     └─ 运行环境、设计与实现约束                                  │
│                                                                 │
│  3. 系统特性 (System Features)                                  │
│     ├─ 功能需求详细规格 (输入/处理/输出)                         │
│     ├─ 用例规约 (前置条件/主流程/异常/后置)                      │
│     └─ 状态图、时序图、类图                                      │
│                                                                 │
│  4. 外部接口 (External Interfaces)                              │
│     ├─ 用户界面、硬件接口                                        │
│     └─ 软件接口、通信接口                                        │
│                                                                 │
│  5. 非功能需求 (Non-functional Requirements)                    │
│     ├─ 性能需求、安全需求                                        │
│     ├─ 软件质量属性 (可靠性/可维护性/可移植性)                   │
│     └─ 业务规则、数据库需求                                      │
│                                                                 │
│  6. 验收标准 (Acceptance Criteria)                              │
│     └─ 测试场景、通过标准                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### SRS示例：推荐引擎模块

```markdown
# 智能题库系统 SRS

## 1. 引言
### 1.1 目的
本文档定义智能推荐引擎的技术规格，供开发和测试团队使用。

### 1.2 范围
覆盖个性化推荐、错题分析、学习路径规划三个核心模块。

## 2. 总体描述
### 2.1 运行环境
- 服务端: Kubernetes集群
- 数据库: PostgreSQL 14 + Redis 7
- ML服务: Python 3.10 + TensorFlow 2.x

## 3. 系统特性

### 3.1 推荐算法服务 (SF-001)

#### 3.1.1 功能描述
基于协同过滤 + 知识图谱的混合推荐算法。

#### 3.1.2 输入
```json
{
  "user_id": "string",
  "subject": "math|physics|chemistry",
  "session_length": 20,
  "difficulty_preference": "easy|medium|hard"
}
```

#### 3.1.3 处理流程
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 获取用户画像 │───>│ 召回候选题目 │───>│ 精排算法     │
│             │    │ (CF + 知识)  │    │ (XGBoost)   │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             │
                                             ▼
                                      ┌─────────────┐
                                      │ 多样性重排   │
                                      │ (MMR算法)   │
                                      └──────┬──────┘
                                             │
                                             ▼
                                      ┌─────────────┐
                                      │ 返回Top-N   │
                                      └─────────────┘
```

#### 3.1.4 输出
```json
{
  "recommendations": [
    {
      "question_id": "Q12345",
      "confidence": 0.85,
      "reason": "薄弱知识点: 二次函数"
    }
  ],
  "estimated_time": 30
}
```

#### 3.1.5 性能要求
- 响应时间: P99 < 500ms
- QPS: 10000
- 缓存命中率: > 80%

## 4. 外部接口

### 4.1 API接口
| 端点 | 方法 | 描述 |
|------|------|------|
| /api/v1/recommend | POST | 获取推荐题目 |
| /api/v1/progress | GET | 获取学习进度 |

### 4.2 数据模型
```sql
-- 用户画像表
CREATE TABLE user_profile (
    user_id UUID PRIMARY KEY,
    knowledge_state JSONB,  -- 知识点掌握度
    learning_path JSONB,    -- 学习路径
    updated_at TIMESTAMP
);

-- 题目知识图谱表
CREATE TABLE question_kg (
    question_id UUID PRIMARY KEY,
    knowledge_points UUID[],
    difficulty FLOAT,
    bloom_level INT
);
```

## 5. 非功能需求

### 5.1 安全需求
- 用户数据加密存储 (AES-256)
- API请求鉴权 (JWT)
- SQL注入防护

### 5.2 可靠性
- 服务降级: 推荐服务故障时返回热门题目
- 熔断机制: 错误率 > 50% 时触发

## 6. 验收标准
- AC-01: 1000并发下响应时间 < 500ms
- AC-02: 推荐准确率 (CTR) > 30%
- AC-03: 系统可用性 > 99.9%
```

### 文档间的信息流转

| 信息类型 | BRD → PRD | PRD → SRS |
|----------|-----------|-----------|
| 商业目标 | 转化为产品目标 | 转化为技术指标 |
| 用户需求 | 细化为用户故事 | 分解为功能点 |
| 功能范围 | 确定MVP边界 | 定义实现细节 |
| 成功标准 | 定义产品指标 | 定义技术SLA |

## 敏捷环境下的文档实践

在敏捷开发中，文档形式更加灵活：

```
传统模式 vs 敏捷模式:

传统:                    敏捷:
┌──────────┐           ┌──────────┐
│ 厚重BRD  │           │ 一页商业画布│
│ (50页)   │           │ (1页)     │
└──────────┘           └──────────┘
     │                      │
     ▼                      ▼
┌──────────┐           ┌──────────┐
│ 详细PRD  │           │ 用户故事地图│
│ (100页)  │           │ + 原型     │
└──────────┘           └──────────┘
     │                      │
     ▼                      ▼
┌──────────┐           ┌──────────┐
│ 完整SRS  │           │ 活文档     │
│ (技术)   │           │ (代码+测试)│
└──────────┘           └──────────┘
```

### 轻量级替代方案

| 传统文档 | 敏捷替代 | 工具 |
|----------|----------|------|
| BRD | 商业画布、OKR | Miro、Notion |
| PRD | 用户故事地图、原型 | Jira、Figma |
| SRS | API文档、自动化测试 | Swagger、Cucumber |

## 面试要点

### Q1: BRD和PRD的核心区别是什么？

**答案**：BRD回答"为什么做"，面向管理层，关注商业价值；PRD回答"做什么"，面向执行团队，关注功能细节。BRD是立项依据，PRD是执行蓝图。

### Q2: PRD中用户故事怎么写？

**答案**：标准格式"作为[用户角色]，我想[功能]，以便[价值]"。好的用户故事符合INVEST原则：独立、可协商、有价值、可估算、短小、可测试。

### Q3: 敏捷开发中还需要PRD吗？

**答案**：需要，但形式更轻量。用用户故事地图替代详细PRD，保持"刚好足够"的文档。核心需求仍需书面记录，避免口头传递造成误解。

### Q4: SRS中的非功能需求包括哪些？

**答案**：性能、安全、可靠性、可用性、可维护性、可扩展性、兼容性等。通常用具体的量化指标描述，如"响应时间 < 200ms"。
